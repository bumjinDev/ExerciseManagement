## 1. 개요

### 1.1. 문서 목적

이 문서는 팀 대항 등수형 챌린지 서비스에서 구현할 기능 요구사항을 정의한다. 기획서가 정한 핵심 규칙(단일 종목 인스턴스, 사회적 검증과 사진-수치 바인딩, 개인·팀 이중 보상, 윈도우 종료 시 단일 정산)을 구현 직전 수준까지 구체화한다.

각 기능은 네 부분으로 구성한다. 기능 개요에서 그 기능이 무엇을 왜 하는지 서술하고, 입출력 데이터 모델에서 다루는 데이터 구조를 정리한다. 사용자 시나리오에는 시간 순서의 흐름을 담되, 흐름보다 데이터 구조가 중심인 기능은 이 자리를 데이터 모델로 대신한다. 기능 요구사항에는 시스템이 지켜야 할 규칙과 제약을 둔다.

기획서가 설계 단계로 미룬 항목과 기획서에 근거가 없는 정량·정책 항목은 이 단계에서 확정하지 않고 미확정으로 표기한다. 입출력 데이터 모델의 구체 타입과 식별자 형식, 응답 시간·처리량 같은 정량 목표, 인증·권한·암호화 정책이 여기에 해당한다.

### 1.2. 검증 책임 분담 설계

이 서비스는 디바이스 자동 측정이라는 신뢰 근거 없이 수행을 검증한다. 그래서 검증을 두 층으로 나누고 각 층의 판정 주체와 책임을 분리한다. 시스템이 기계로 판정할 수 있는 부분과 사람만 판정할 수 있는 부분이 다르기 때문이다.

| 구분 | 시스템(기계)의 역할 | 팀원(사람)의 역할 |
|------|------------------|------------------|
| **핵심 책임** | 위·변조와 빈도 위반을 기계로 차단한다 | 인증샷의 진위를 사람으로 판정한다 |
| **검사 대상** | - 사진 해시 중복(같은 사진 재사용)<br>- 사진 메타데이터 정합성<br>- 인증 시간 윈도우(하루 횟수 상한, 인증 주기) | - 인증샷이 실제 그 수행을 담았는지의 진위 |
| **점수에 미치는 영향** | - 검증 통과 시 제출을 확인 대기로 진입시킨다<br>- 이 시점에는 점수에 반영하지 않는다 | - 확인하면 그 제출의 볼륨을 제출자 개인 누적과 소속 팀 누적에 함께 반영한다<br>- 반려하면 제출을 무효로 만든다 |
| **다루지 않는 것** | - 사진이 담은 운동이 그 종목인지의 내용 판별 | - 위·변조와 빈도 위반의 기계 판정 |

**설계 의의:** 점수를 움직이는 행위는 제출이 아니라 팀원의 확인이다. 시스템은 점수 반영 앞에 검증 게이트를 한 겹 두어, 사람이 판정하기 전에 위·변조와 빈도 위반을 먼저 걸러낸다. 기계가 닫을 수 있는 영역과 사람만 닫을 수 있는 영역을 나눠 맡기는 이 구조가, 자동 측정이 안 되는 종목까지 챌린지 대상으로 끌어들이는 근거가 된다.

## 2. 기능 요구사항 명세

### F001 - 챌린지 등록과 성립 검증

#### 1. 기능 개요

등록자가 새 챌린지를 만든다. 챌린지는 부위 카테고리 안의 단일 종목 하나로 묶이고, 같은 종목 안에서는 무게가 바로 비교되므로 종목 사이의 무게 환산 규칙이 필요 없다. 등록자는 종목과 함께 팀 구성, 상금 받을 팀 수, 참가 예치금, 모집·수행 기간, 확인 윈도우 길이, 인증 빈도를 정한다. 인증 빈도는 팀 볼륨 집계를 위한 규칙이며, 참가자가 참가 신청 때 정하는 개인 목표와는 별개다. 시스템은 등록 시점에 성립 제약을 검사하고, 검사를 통과한 챌린지를 모집 상태로 공개한다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 등록자가 등록 화면에서 입력한다.

```json
{
  "카테고리": "시스템이 미리 정한 부위 카테고리 목록에서 고른 단일 카테고리(예: 가슴, 등, 어깨, 팔, 하체)",
  "종목": "선택한 카테고리 안의 종목 목록에서 고른 단일 종목(자유 입력 불가)",
  "팀_정원": "한 팀의 인원 수",
  "팀_수": "챌린지를 구성하는 팀의 수",
  "상금_받을_팀_수": "전체 팀 수보다 작은 값",
  "참가_예치금": "참가 신청이 완료되는 시점에 참가자 예치 잔액에서 차감되는 액수. 0보다 큰 값",
  "모집_기간": "모집 시작과 종료의 날짜와 시각",
  "수행_기간": "수행 시작과 마감의 날짜와 시각",
  "확인_윈도우_길이": "제출 완전 마감 이후 열리는 확인 유예 기간의 길이. F008이 제출 완전 마감 시각(수행 마감 + 소급 유예 24시간)에 이 길이를 더해 윈도우 종료 시점을 정한다",
  "인증_빈도": {
    "하루_인증_횟수_상한": "하루에 인정하는 인증 횟수의 상한(필수, 무제한 불가). 팀 볼륨 도배 방지선",
    "인증_주기": {
      "주기_방식": "'며칠에 한 번' 또는 '주 며칠' 중 어느 방식으로 주기를 재는지",
      "주기_간격": "선택한 방식의 간격을 나타내는 숫자값(예: '며칠에 한 번'이면 며칠, '주 며칠'이면 주당 일수)"
    }
  }
}
```

**출력 데이터:** 성립 검증을 통과한 챌린지를 모집 상태로 공개한다.

```json
{
  "챌린지": {
    "챌린지_ID": "시스템이 부여하는 챌린지 식별 번호. 이후 참가 신청, 기준 측정, 제출 등에서 이 챌린지를 가리키는 데 쓴다",
    "카테고리": "선택된 단일 카테고리",
    "종목": "선택된 단일 종목",
    "목표_참가_인원": "팀 정원 × 팀 수로 산출된 값",
    "상태": "모집"
  }
}
```

#### 3. 기능 요구사항

- 카테고리와 종목은 시스템이 미리 정한 목록 안에서만 선택한다. 등록자는 부위 카테고리를 먼저 고르고 그 카테고리 안의 종목 목록에서 단일 종목을 고른다. 종목 집합이 닫혀 있어야 같은 운동이 여러 이름으로 갈라지지 않고 무게 비교와 종목별 집계가 닫힌다.
- 목표 참가 인원은 팀 정원과 팀 수의 곱으로 산출한다. 이 값으로 모집하면 시작 시 모든 팀 인원이 같고, 정원으로 나누어떨어지지 않는 나머지 인원이 생기지 않는다.
- 상금을 받을 팀 수는 전체 팀 수보다 작아야 하며, 시스템이 이 제약을 강제한다. 상금을 못 받는 팀이 있어야 순위 경계가 당락을 가른다.
- 참가 예치금 액수는 등록자가 챌린지마다 정하며 0보다 큰 값이어야 한다. 참가 신청(F002)이 완료되는 시점에 이 액수가 참가자 예치 잔액에서 차감되며, 참가자는 액수를 입력하지 않는다. 같은 챌린지의 참가자 전원이 같은 액수를 건다. 액수를 참가자가 각자 정하지 않는 이유는, 미달자 차감분이 팀 상금풀의 재원으로 들어가는 구조에서 예치 액수가 참가자마다 다르면 상금풀 크기가 참가자의 선택에 좌우되고 개인 차감의 무게도 참가자마다 달라지기 때문이다.
- 하루 인증 횟수 상한은 반드시 존재해야 한다. 상한이 없으면 한 사람이 같은 운동을 도배해 팀 볼륨을 무한정 부풀릴 수 있어, 시스템이 하루 무제한을 허용하지 않는다. 이 상한은 참가자가 F002에서 정하는 개인 목표 빈도가 넘을 수 없는 한계 역할도 겸한다. 개인 목표 빈도는 이 상한 안에서만 인증을 올릴 수 있다.
- 인증 빈도와 주기의 값 자체는 등록자가 정한다. 이 빈도는 하루 인증 횟수 상한과 인증 주기로 이뤄지며, 팀 볼륨 집계에만 쓰이고 개인 목표와는 별개 트랙이다.
- 모집 기간과 수행 기간은 시작과 끝을 날짜와 시각까지 잡는다. 수행 기간은 하나의 연속 구간이며 날짜별로 쪼개지 않는다.
- 확인 윈도우 길이는 챌린지마다 등록 시 개별로 정한다. F008은 제출 완전 마감 시각(수행 마감 + 소급 유예 24시간)에 이 길이를 더한 시점을 확인 윈도우 종료로 삼는다.

### F002 - 챌린지 참가 신청

#### 1. 기능 개요

참가자가 모집 중인 챌린지에 신청하고 예치금을 건다. 신청할 때 참가자는 자기 개인 목표도 함께 정한다. 개인 목표는 등록자가 아니라 참가자 본인이 정하며, 이 값이 있어야 신청이 성립한다. 모집 기간에는 신청만 받고 참가자에게 아직 팀이 없다. 목표 인원이 차면 팀 편성이 실행되고 챌린지가 시작한다. 모집 기간이 끝날 때까지 목표 인원을 못 채우면 챌린지는 무산되고 예치금을 전액 반환한다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 참가자가 신청과 함께 개인 목표를 넣는다. 강도 자체는 입력하지 않는다. 시스템이 확정된 개인 목표 기준값에 계수를 곱해 강도 하한을 산출하므로, 참가자가 넣는 값은 그 계수와 목표 빈도다.

```json
{
  "챌린지_ID": "신청 대상 챌린지의 식별 번호(F001 등록 응답에서 받은 값)",
  "개인_목표": {
    "강도_계수": "개인 목표 기준값에 곱해 개인 강도 하한을 정하는 계수. 1 이상, 미입력 시 1",
    "목표_빈도": {
      "설명": "강도 하한을 채운 제출을 얼마나 자주 올릴지",
      "주기_방식": "'며칠에 한 번' 또는 '주 며칠' 중 어느 방식",
      "주기_간격": "그 방식의 간격을 나타내는 숫자값"
    }
  }
}
```

#### 3. 사용자 시나리오

참가 신청은 신청 진입과 과거 값 확정, 개인 목표 설정, 예치 차감, 모집 결과 분기 순으로 이어진다.

**3.1. 신청 진입과 과거 값 확정**
참가자가 챌린지 상세에서 참여를 결정하고 신청을 시작한다. 시스템이 먼저 그 종목의 최근 30일 확인 완료 기록을 조회한다. 윈도우 내 기록이 있으면 그 제출들에서 편성 실력과 개인 목표 기준값을 바로 확정하고, 없으면 그 자리에서 기준 측정 인증을 받아 두 값을 확정한다. 기준 측정의 검증은 F003이 담당한다.

**3.2. 개인 목표 설정**
확정된 개인 목표 기준값 위에서 참가자가 자기 개인 목표를 정한다. 참가자는 곱해질 기준값을 보면서 강도 계수를 얹고 목표 빈도를 정한다. 개인 목표까지 입력되면 참가자는 팀에 배정되지 않은 상태로 모집 명단에 들어간다.

**3.3. 예치 차감**
참가 신청과 함께 그 챌린지의 참가 예치금(F001 등록값)이 참가자의 예치 잔액에서 차감된다. 예치금 충전은 F010이 담당한다.

**3.4. 모집 결과 분기**
목표 인원이 차면 시스템이 편성을 실행하고 챌린지를 시작한다. 모집 기간이 끝날 때까지 목표 인원이 차지 않으면 챌린지는 무산되고, 이미 예치한 참가자에게 예치금을 전액 반환한다.

#### 4. 기능 요구사항

- 참가 신청은 모집 기간 안에서만 받는다.
- 신청 진입 시 시스템이 그 종목의 최근 30일 확인 완료 기록을 조회해 편성 실력과 개인 목표 기준값을 확정한다. 윈도우 내 기록이 없으면 기준 측정(F003)을 신청 절차 안에서 받아 두 값을 확정한다. 개인 목표 기준값 확정은 개인 목표 입력에 선행해야 한다. 개인 목표 계수가 곱해질 기준값이 그 시점에 있어야 하기 때문이다.
- 신청 시 참가자는 개인 목표(강도 계수, 목표 빈도)를 함께 넣어야 하며, 이 값이 있어야 신청이 성립한다.
- 강도 계수는 1 이상이어야 하고, 미입력 시 1로 처리한다. 계수가 1이면 개인 목표 기준값을 그대로 강도 하한으로 쓰고, 1보다 크면 그만큼 높은 강도를 목표로 건다. 1 미만은 허용하지 않아 자기 기준값보다 쉬운 목표를 잡지 못하게 한다.
- 개인 강도 자체는 참가자가 신고하지 않는다. 시스템이 개인 목표 기준값(F003)에 계수를 곱해 개인 강도 하한을 산출한다. 강도를 참가자가 신고하지 않으므로 강도를 낮게 신고해 목표를 쉽게 만드는 조작이 생기지 않는다.
- 개인 목표 빈도는 등록자가 정한 하루 인증 횟수 상한을 넘을 수 없다. 이 상한 위로는 인증을 올리는 것 자체가 막혀 있다.
- 신청 시 그 챌린지의 참가 예치금(F001 등록값)을 참가자 예치 잔액에서 차감한다. 액수는 참가자가 입력하는 값이 아니다.
- 목표 인원이 도달하는 시점에 팀 편성(F004)을 실행한다.
- 모집 기간 종료 시 목표 인원 미달이면 챌린지를 무산하고 예치금을 전액 반환한다.

### F003 - 기준 측정 인증

#### 1. 기능 개요

팀 편성과 개인 목표 강도 하한 산출에는 각 참가자의 그 종목 수행 값이 입력으로 들어간다. 두 쓰임 모두 같은 값을 쓴다. 신청 시점 기준 최근 30일 안의 확인 완료 제출 볼륨의 1회 평균이다.

1회 평균을 쓰는 이유는 값을 서로 비교하기 위해서다. 편성 알고리즘은 기록 보유자의 값과 신규 참가자의 기준 측정값을 한 줄에 세워 더하고 뺀다. 기준 측정값은 1회 수행 수준이므로 기록 보유자의 값도 1회 수행 수준이어야 두 경로의 숫자가 같은 종류가 된다. 누적 총합은 제출 횟수가 섞여 들어가 같은 실력의 참가자를 이력 길이에 따라 다른 값으로 만들므로 쓰지 않는다. 과거의 제출 횟수는 그때 참가한 챌린지의 빈도 규칙에 따라 달라지는 값이고, 이번 챌린지의 제출 횟수는 이번 챌린지의 빈도 규칙과 예치금이 정하므로 횟수는 편성 신호에서 뺀다.

윈도우를 두는 이유는 오래된 기록이 현재 수행 수준을 대표하지 못하기 때문이다. 개인 목표 기준값에도 같은 윈도우를 적용해, 오래 쉰 참가자의 목표 하한이 과거 전성기 평균으로 잡히는 문제를 막는다. 신청 시점 기준 최근 30일 안에 그 종목 확인 완료 기록이 없는 참가자는 그 종목으로 기준 측정 인증을 한 번 올리고 그 1회 볼륨을 두 값으로 쓴다. 사용자에게 기준 측정을 요청하는 시점은 F002 참가 신청 절차 안이다.

기준 측정은 일반 제출과 달리 팀원 확인을 거치지 않고 시스템 기계 검증만 거친다. 기준 측정은 팀 배정 전이라 확인할 팀원이 없고, 자기 수행 수준을 남기는 측정이라 다른 참가자와 겨루는 값이 아니다. 측정값을 높이면 개인 강도 하한이 올라가 자기 목표만 어려워지고 편성에서 강팀으로 취급돼 오히려 불리하므로, 사람 확인으로 막을 부정 유인이 약하다. 사람 확인만 빠질 뿐 해시 중복과 메타데이터 정합성 검사는 그대로 거치므로 검증 모델 안에 있다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 그 종목의 최근 30일 내 확인 완료 기록이 없는 참가자가 참가 신청 절차 안에서 올린다. 카테고리와 종목은 참가자가 자유 선택하는 값이 아니라 대상 챌린지(F001에서 단일 종목으로 고정됨)에서 결정된다. 어느 종목의 측정인지가 그 산출 값이 어느 종목에 속하는지를 정한다.

```json
{
  "카테고리": "대상 챌린지의 부위 카테고리(예: 가슴, 등, 어깨, 팔, 하체)",
  "종목": "대상 챌린지의 단일 종목. 이 값이 산출 값이 속하는 종목이 된다",
  "무게": "기준 측정 수행의 무게",
  "횟수": "기준 측정 수행의 횟수",
  "인증샷": "수행 인증 이미지"
}
```

**출력 데이터:** 검증을 거친 두 값. 편성 실력은 팀 편성(F004)의 입력이고, 개인 목표 기준값은 개인 목표 강도 하한(F002 계수를 곱하는 기준값)의 입력이다. 두 값은 같은 계산에서 나오는 동일한 값이며, 쓰이는 자리를 구분하기 위해 명칭만 나눈다.

```json
{
  "편성_실력": "신청 시점 기준 최근 30일 확인 완료 제출 볼륨의 1회 평균. 윈도우 내 기록이 없으면 기준 측정 1회 볼륨",
  "개인_목표_기준값": "편성 실력과 동일한 값. 쓰이는 자리(개인 강도 하한의 기준)만 다르다"
}
```

#### 3. 기능 요구사항

- 기준 측정 인증은 신청 시점 기준 그 종목의 최근 30일 내 확인 완료 기록이 없는 참가자에게 요구한다. 과거에 기록이나 측정이 있어도 윈도우를 벗어났으면 다시 측정한다. 사용자에게 요청하는 시점은 F002 참가 신청 절차 안이다.
- 최근 30일 내 그 종목 기록이 있는 참가자는 측정을 건너뛰고 윈도우 안의 확인 완료 제출에서 편성 실력과 개인 목표 기준값을 산출한다.
- 기준 측정은 팀원 확인 없이 시스템 기계 검증만 거친다. F005 제출과 동일하게 해시 중복과 메타데이터 정합성을 검사한다.
- 편성 실력과 개인 목표 기준값은 동일한 값이다. 신청 시점 기준 최근 30일 확인 완료 제출 볼륨의 1회 평균이며, 앞의 것은 팀 편성 입력으로, 뒤의 것은 개인 목표 강도 하한의 기준값으로 쓴다. 누적 총합을 쓰지 않는 이유는 총합에 제출 횟수가 섞여 같은 실력의 참가자가 이력 길이에 따라 다른 값을 받기 때문이고, 신규 참가자의 기준 측정값(1회 수행 수준)과 같은 종류의 숫자가 되도록 1회 평균으로 둔다.
- 집계 대상은 신청 시점부터 거슬러 30일 안의 확인 완료 제출로 한정한다. 윈도우 안에서는 가중을 주거나 일부만 골라 쓰지 않고 단순 평균한다. 윈도우 길이 30일은 잠정 설정값이다. 짧으면 최신이지만 표본이 적고 길면 표본이 많지만 낡는 절충이 있으며, 적정 길이는 운영 데이터 축적 후 재검토한다.
- 기준 측정은 한 번의 신청 절차 안에서 1회만 받는다. 반복 측정으로 대표값을 뽑지 않으며, 그 1회 볼륨이 편성 실력과 개인 목표 기준값이 된다.
- 측정값을 낮게 남기려는 조작(약하게 인증)은 이 검증 모델로 완전히 막지 못한다. 디바이스 생체 측정이나 실시간 지도가 없어 실제 수행 강도를 기계 검증만으로 판별할 수 없는 구조적 한계이며, 개인 프로젝트 범위의 한계로 둔다.

### F004 - 팀 편성

#### 1. 기능 개요

목표 인원이 차면 시스템이 참가자 전체를 한꺼번에 팀으로 나눈다. 참가자는 팀을 직접 고르지 않는다. 편성에는 각 참가자의 편성 실력, 곧 그 종목 최근 30일 확인 완료 제출 볼륨의 1회 평균을 입력으로 쓴다.

팀 대항의 공정성은 두 가지 균형에서 나온다. 하나는 팀 사이 편성 실력 총합이 비슷한 실력 합 균형이고, 다른 하나는 팀 안 실력 구성이 한쪽으로 쏠리지 않는 실력 분포 균형이다. 실력 합이 같아도 고실력자가 한 팀에 몰리면 볼륨 경쟁에서 성격이 달라지므로 두 균형을 함께 맞춰야 한다.

두 균형은 서로 상충한다. 분포를 고치는 교환은 편성 실력이 다른 두 사람을 맞바꾸는 것이라 그 실력 차이만큼 팀 합이 함께 움직이고, 일반적인 참가자 구성에서 두 균형을 동시에 완전하게 만들 수 없다. 그래서 편성은 두 균형을 하나의 종합 점수로 합쳐 최적화하지 않고 역할을 비대칭으로 나눈다. 실력 합 균형은 넘으면 안 되는 제약으로 두고, 실력 분포 균형은 그 제약 안에서 최소화하는 목표로 둔다. 두 편차는 단위도 달라(합 편차는 볼륨 원단위, 분포 편차는 볼륨 제곱 단위) 하나의 점수로 합치려면 정규화가 필요한데, 정규화 분모 선택이 결과를 흔드는 문제가 검증 과정에서 확인되어 단일 점수 방식은 쓰지 않는다.

#### 2. 처리 데이터 모델

**처리 입력:** 외부 요청 바디가 아니다. F002에서 인원이 찬 순간 편성 연산이 자기 데이터에서 읽어들이는 내부 상태다. 참가자 목록은 그 챌린지 신청자 전체이고, 팀 정원과 팀 수는 F001 등록값이며, 합 상한 r은 시스템 설정값이다.

```json
{
  "챌린지_ID": "편성 대상 챌린지 식별 번호",
  "참가자_목록": [
    { "참가자": "참가자 식별", "편성_실력": "최근 30일 확인 완료 제출 볼륨의 1회 평균 또는 기준 측정 1회 볼륨" }
  ],
  "팀_정원": "F001 등록값. 한 팀의 인원 수",
  "팀_수": "F001 등록값. 챌린지를 구성하는 팀의 수",
  "합_상한_r": "합 편차 허용 비율(%). 시스템 설정값이며 잠정값. 확정 방법은 아래 잠정 설정값 항목에 있다"
}
```

**출력 데이터:** 팀 배정 결과. 모든 팀의 인원이 같다. 배정이 끝나면 챌린지가 시작 상태로 전이한다.

```json
{
  "팀_배정": [
    { "팀": "팀 식별", "팀원_목록": ["배정된 참가자들"] }
  ]
}
```

#### 3. 기능 요구사항

- 편성 입력으로 쓰는 실력은 참가자가 스스로 고르는 등급이 아니라 행동 데이터에서 잡은 편성 실력이다. 자기 신고 등급은 검증할 수단이 없고 실력을 낮게 신고할 유인이 있어 쓰지 않는다.
- 챌린지가 단일 종목이므로 편성에는 그 종목의 편성 실력만 쓴다.
- 편성은 외부 요청으로 호출되지 않는다. F002에서 목표 인원이 차는 것이 유일한 실행 조건이며, 인원이 차는 순간 시스템이 내부 데이터로 실행한다.
- 인원 충족 판정과 편성 시작은 단일 실행으로 묶여야 한다. 마지막 자리를 두고 여러 신청이 동시에 들어와도 정원을 초과하거나 편성이 두 번 실행되지 않아야 한다.
- 참가자는 팀을 직접 고를 수 없고 배정은 시스템이 한다. 목표 인원이 팀 정원과 팀 수의 곱이므로 나머지 인원 없이 모든 팀 인원이 같다.

**평가 기준**

- 실력 합 편차는 최대 팀 합에서 최소 팀 합을 뺀 값이다. 합 규칙은 `합 편차 ÷ 전체 팀 합의 평균 ≤ r%`이며, 절차 내부에서는 전체 팀 합의 평균 × r%로 환산한 허용폭 하나로 판정한다. 절대값이 아니라 비율에 상한을 거는 이유는 같은 격차라도 챌린지 규모에 따라 무게가 달라지기 때문이며, 비율로 두면 상한 r 하나가 종목과 규모에 무관하게 모든 챌린지에서 같은 뜻으로 작동한다.
- 실력 분포 편차는 팀별 편성 실력 분산 가운데 분산 최대 팀과 최소 팀의 차이다. 이 지표는 합 제약이 지켜지는 배정에서만 의미가 있으므로 단독으로 쓰지 않는다.
- 두 편차를 하나의 종합 점수로 합치지 않는다. 합 편차는 허용폭을 넘으면 안 되는 제약이고, 분포 편차는 그 제약 안에서 최소화하는 목표다.
- 분포 편차에 고정 합격선을 두지 않는다. 도달 가능한 최소 분포 편차는 참가자 구성이 정하므로, 고정선은 균등한 풀에서는 무의미한 기준이 되고 이상치가 낀 풀에서는 달성 불가능한 기준이 된다.

**편성 로직**

편성은 다음 네 단계 순서로 실행된다.

1. 준비. 그 챌린지 신청자 전체의 편성 실력을 모으고, 전체 팀 합의 평균과 허용폭(전체 팀 합의 평균 × r%)을 계산한다. 팀 정원과 팀 수는 F001 등록값이고, 참가자 수는 정원과 팀 수의 곱이라 팀마다 정원대로 정확히 나뉜다.
2. 시작 배정 생성. 시작 배정 두 개를 만든다. 교대 방향 배정은 참가자를 편성 실력 내림차순으로 세워 라운드마다 방향을 뒤집으며 팀에 한 명씩 배정하고, 내림차순 그리디는 큰 값부터 하나씩 자리가 남은 팀 가운데 현재 합이 가장 작은 팀에 배정한다. 두 방식 모두 배정 과정 자체가 정원을 지킨다. 시작점을 두 개 두는 것은 한쪽이 더 나아질 수 없는 지점에 갇혀도 다른 쪽이 더 나은 배정에 닿게 하려는 것이다. 3단계의 교환 반복이 개선 교환만 실행해 시작점마다 도착점이 달라질 수 있고, 우세한 시작점이 참가자 구성과 합 상한에 따라 뒤집히는 것이 검증 과정에서 확인되어 둘 다 실행하고 좋은 결과를 채택한다.
3. 교환 반복. 시작 배정마다 독립으로 실행한다. 서로 다른 팀에 속한 두 사람의 모든 짝을 후보로 만들고, 짝마다 교환 후의 합 편차와 분포 편차를 계산한다. 합 필터는 교환 후 합 편차가 허용폭을 넘는 짝을 버리되, 현재 배정이 이미 허용폭 밖이면(이상치 풀) 합 편차를 현재보다 키우지 않는 짝을 남긴다. 필터를 통과한 짝 가운데 분포 편차를 가장 많이 줄이는 하나를 실제로 교환하며, 감소량 동률이면 참가자 식별자가 앞선 짝을 잡는다. 필터를 통과하면서 분포 편차를 줄이는 짝이 하나도 없으면 반복이 종료된다. 외부의 반복 횟수 제한은 두지 않는다.
4. 채택과 전이. 두 시작점의 결과 가운데 허용폭을 지킨 결과가 하나라도 있으면 지킨 결과들만 후보로 남기고, 그중 분포 편차가 낮은 쪽을, 동률이면 합 편차가 낮은 쪽을 채택한다. 두 결과 모두 허용폭을 어긴 경우에는 합 편차가 낮은 쪽을, 동률이면 분포 편차가 낮은 쪽을 채택한다. 확정 후 챌린지를 시작 상태로 전이시킨다.

**보장 사항**

- 모든 팀 인원이 같고 나머지 인원이 없다. 변형이 1대1 맞교환뿐이라 반복 중에도 정원이 유지된다.
- 편성은 실패로 처리되지 않는다. 목표 인원이 찬 챌린지의 편성은 반드시 완료되며(F002), 허용폭 달성이 불가능한 풀에서도 그 풀에서 가능한 최선의 합 격차를 유지한 배정을 낸다.
- 반드시 종료한다. 매 교환이 분포 편차를 줄이기만 하고 배정의 가짓수가 유한해, 같은 배정으로 되돌아오는 왕복이 일어나지 않는다.
- 결정적이다. 동률 규칙까지 정의되어 같은 입력이면 항상 같은 출력이 나온다.
- 멈추는 지점은 1대1 맞교환이라는 변형 범위 안에서의 최소일 뿐, 전체에서 가장 낮은 값까지는 보장하지 않는다.

**잠정 설정값과 미확정 경계**

- 합 상한 r은 잠정 설정값이다. 여러 참가자 구성 모양에 대해 r별 도달 분포 편차의 절충 곡선을 만들고, 더 풀어도 분포가 거의 좋아지지 않는 꺾임점 근처로 정한다. (잠정)
- 반복 1회의 후보 수는 K(K−1)/2 × N²(K는 팀 수, N은 팀 정원)로 팀 수와 인원에 따라 커지는 CPU 바운드 구간이다. 분산 최대 팀 또는 최소 팀이 낀 팀 짝으로만 후보를 줄이는 방법은 결과가 변하지 않음이 증명되어 있어 구현에서 선택적으로 붙일 수 있다. 교환 하나는 관련된 두 팀의 분산만 바꾸므로, 최대 팀도 최소 팀도 끼지 않은 팀 짝의 교환은 최대 분산을 낮추지도 최소 분산을 높이지도 못해 분포 편차를 줄일 수 없고, 이런 짝은 버려도 채택 결과가 같다. 반대로 최대 팀이나 최소 팀이 한쪽만 낀 팀 짝은 분포 편차를 가장 많이 줄이는 교환이 될 수 있으므로, 후보를 (최대, 최소) 한 짝으로 좁히는 것은 결과를 바꿔 허용되지 않는다. 증분 재계산과 병렬 분할 같은 구현 최적화는 구현 이후 계측을 근거로 정한다. 위 절차 정의는 여기서 종결한다. (미확정)

### F005 - 인증 제출과 시스템 검증

#### 1. 기능 개요

참가자가 무게와 횟수를 인증샷과 함께 올린다. 제출할 때 운동을 수행한 날짜를 함께 지정하며, 그 날짜가 지났더라도 수행 시점으로부터 24시간 안에는 그 날짜로 등록할 수 있다(소급 등록 유예). 제출은 올라온 즉시 점수가 되지 않고 확인 대기로 들어간다. 점수 반영 앞에서 시스템이 먼저 기계 검증을 수행한다. 해시 중복으로 같은 사진 재사용을 거르고, 메타데이터 정합성을 검사하며, 지정한 수행 날짜가 소급 유예 안인지 확인하고, 시간 윈도우로 인증 빈도를 강제한다. 이 검증을 통과한 제출만 확인 대기 상태로 남아 팀원 확인을 기다린다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 참가자가 인증 제출 화면에서 올린다.

```json
{
  "챌린지_ID": "제출 대상 챌린지의 식별 번호. 종목은 이 챌린지에서 결정된다",
  "무게": "수행 무게",
  "횟수": "수행 횟수",
  "수행_날짜": "운동을 수행한 날짜. 제출 시점 기준 24시간 안의 날짜만 허용하며, 미래 날짜는 받지 않는다",
  "인증샷": "수행 인증 이미지"
}
```

**출력 데이터:** 검증 결과에 따라 확인 대기 제출로 남거나 거부된다. 남는 제출은 어느 챌린지의 어느 수행 날짜에 속하는지와 등록 시점이 함께 기록된다.

```json
{
  "제출_결과": {
    "상태": "확인 대기 또는 검증 실패 거부",
    "챌린지_ID": "이 제출이 속한 챌린지",
    "연결_날짜": "참가자가 지정한 수행 날짜. 제출이 이 날짜에 연결되어 저장되며, 하루 상한과 인증 주기 판정의 기준이다",
    "등록_시점": "참가자가 인증을 올린 시점",
    "거부_사유": "해시 중복 / 메타데이터 불일치 / 소급 유예 초과 / 빈도 위반 중 해당 항목(거부 시)"
  }
}
```

#### 3. 기능 요구사항

- 한 제출은 무게, 횟수, 수행 날짜, 인증샷 네 가지로 이뤄진다. 인증샷이 검증 대상이고 무게와 횟수가 수치이며, 숫자만 올리는 신고는 받지 않는다.
- 참가자는 제출할 때 운동을 수행한 날짜를 지정한다. 지정할 수 있는 날짜는 제출 시점 기준 24시간 안으로 제한하며(소급 등록 유예), 미래 날짜는 받지 않는다. 유예를 24시간으로 제한하는 이유는, 지난 날짜를 자유롭게 지정할 수 있으면 하루에 몰아서 운동하고 여러 날짜로 흩어 등록해 인증 주기를 무력화할 수 있기 때문이다. 24시간이면 하루치 지각 등록은 받아주면서 몰아치기는 막힌다.
- 시스템은 사진 해시 중복을 검사해 같은 사진 재사용을 거른다.
- 시스템은 사진 메타데이터 정합성을 검사한다.
- 시스템은 시간 윈도우로 등록자가 정한 인증 빈도를 강제한다. 하루 횟수 상한을 넘기거나 인증 주기를 어긴 제출은 여기서 거른다. 이 판정은 올린 시점이 아니라 참가자가 지정한 수행 날짜를 기준으로 한다.
- 사진이 실제로 그 운동을 담았는지를 가리는 내용 판별은 이 기능에서 다루지 않는다. 진위 판정은 팀원 확인(F006)이 맡는다.
- 검증을 통과한 제출만 확인 대기 상태로 남는다.
- 각 제출은 어느 챌린지의 어느 수행 날짜에 속하는지 식별되어 저장된다. 참가자가 인증을 올린 등록 시점을 함께 기록하고, 제출은 참가자가 지정한 수행 날짜에 연결한다. 이를 통해 참가자가 언제 무엇을 수행했는지가 날짜 단위로 관리된다.
- 날짜에 연결되어 저장된 인증 내역은 참가자가 날짜 단위로 조회할 수 있어야 한다. 이 날짜별 열람 화면은 내 제출 현황 쪽에 둔다. F005는 제출 시점의 날짜 연결과 시점 기록까지 담당하고, 열람은 이 조회 가능 요구로만 둔다.
- 날짜 안에서 시간대는 별도로 구분하지 않는다.

### F006 - 팀원 확인과 점수 반영

#### 1. 기능 개요

팀원이 확인 대기 제출의 인증샷을 보고 확인하거나 반려한다. 디바이스가 수치를 보증하는 대신 팀원의 확인이 신뢰의 근거가 되며, 점수를 움직이는 것은 제출 자체가 아니라 이 확인 행위다. 확인은 제출 한 건 단위의 처리이고, 확인되면 그 한 건의 볼륨이 제출자 개인 누적과 소속 팀 누적에 더해진다. 반려된 제출은 그 자리에서 무효로 종료되며, 참가자가 다시 올리는 것은 별개의 새 제출 건이다.

#### 2. 사용자 시나리오

제출은 확인 대기에서 시작해 확인 또는 반려를 거치고, 확인이면 증분 반영으로, 반려면 새 제출로 갈린다.

**2.1. 확인 대기**
시스템 검증을 통과한 제출이 확인 대기 상태로 팀원의 확인 큐에 들어간다.

**2.2. 확인 또는 반려**
팀원이 인증샷을 보고 확인하거나 반려한다. 본인이 자기 제출을 확인하는 것은 막는다. 제출이 확인 완료로 넘어가는 데 필요한 팀원 확인 수는 1로 둔다.

**2.3. 증분 반영 또는 새 제출**
확인되면 그 제출 한 건이 확인 완료 상태가 되고, 그 한 건의 볼륨이 개인 누적과 팀 누적에 더해진다. 이미 쌓인 누적에 이번 건의 볼륨을 더하는 증분 반영이며, 개인 기록 전체를 다시 합산하지 않는다. 확인 완료는 종착 상태라 그 제출은 다시 확인 대상이 되지 않는다. 반려되면 그 제출은 무효로 종료되고 확인 완료로 가지 않는다. 참가자가 다시 올리면 기존 건을 되돌리는 것이 아니라 새 제출 건으로 확인 대기부터 다시 시작한다.

#### 3. 기능 요구사항

- 제출이 확인 완료로 넘어가는 데 필요한 팀원 확인 수(정족수)는 1로 둔다.
- 본인이 자기 제출을 확인하는 것을 막는다.
- 확인은 확인 대기 상태의 제출 한 건을 대상으로 하는 일회성 상태 전이다. 확인 행위가 개인 기록 전체를 다시 검사하거나 이미 확인된 제출을 재확인하지 않는다.
- 확인 시 그 제출 한 건의 볼륨을 개인 누적과 팀 누적에 더한다. 같은 한 건의 볼륨이 두 집계에 함께 더해지는 증분 반영이며, 개인 기록 전체를 다시 합산하지 않는다.
- 한 제출의 확인 완료 전이는 한 번만 일어나야 한다. 같은 제출에 확인이 중복으로 들어와도 볼륨이 두 번 가산되지 않아야 한다. 정족수가 1이므로 첫 확인이 확인 완료로 넘긴 뒤에는 그 제출이 더 이상 확인 대상이 아니다.
- 반려된 제출은 그 자리에서 무효로 종료되며 확인 완료로 가지 않는다. 참가자가 다시 올리면 기존 건의 상태를 되돌리는 것이 아니라 새 제출 건으로 확인 대기부터 시작한다.
- 제출 상태 기계는 확인 대기가 유일한 진입 상태이고, 확인 완료·반려·만료는 모두 종착 상태다. 확인 대기는 팀원 확인을 기다리는 상태, 확인 완료는 확인되어 볼륨이 반영된 상태, 반려는 반려로 무효 종료된 상태, 만료는 윈도우 종료까지 확인되지 않은 상태다. 종착에 도달한 제출은 다른 상태로 이동하지 않는다.

### F007 - 순위와 개인 목표 현황 산출

#### 1. 기능 개요

이 기능은 세 가지를 산출한다. 팀끼리 겨루는 팀 순위, 한 팀 안에서 팀원을 기여도로 세우는 팀 내 순위, 참가자 자신의 개인 목표 달성 현황이다. 세 축은 대상 집합과 기준이 서로 다르다.

팀 순위는 팀 사이의 경쟁이며, 팀별 볼륨 누적 합으로 전체 팀을 세운다. 팀 순위에는 상금을 받는 팀과 못 받는 팀을 가르는 순위 경계가 있다. F001에서 상금 받을 팀 수를 전체 팀 수보다 작게 정하므로, 그 수만큼의 상위 팀과 나머지 사이에 경계가 생긴다. 참가자는 자기 팀이 이 경계 위에 있는지 아래에 있는지, 곧 상금권 안인지 밖인지를 확인한다.

팀 내 순위는 한 팀 안에서만 팀원을 세운다. 각 팀원이 그 팀 누적 합에서 차지하는 볼륨 비율이 기준이며, 이 순위가 팀 상금 분배의 근거다. 팀이 받은 상금을 팀원 볼륨 비율로 나누므로 팀 안에서 많이 기여한 팀원이 많이 받는다. 팀 순위가 전체 팀을 대상으로 하는 것과 달리 팀 내 순위는 대상이 한 팀으로 한정된다. 개인 볼륨은 팀 점수의 재료이자 팀 내 순위의 기준이다. 다만 팀을 가로질러 전체 참가자를 볼륨으로 줄 세우는 개인 종합 순위는 두지 않는데, 개인은 볼륨으로 남과 겨루지 않고 자기 목표 달성 여부로만 판정되기 때문이다.

개인 목표 달성 현황은 순위가 아니다. 개인 보상은 목표를 넘겼는지 못 넘겼는지의 달성·미달 판정이라 등수가 성립하지 않는다. 참가자는 남과 겨루는 위치가 아니라 자기가 통과선을 넘었는지와 목표까지 남은 정도를 본다. 같은 제출 데이터가 팀 쪽에서는 볼륨이라는 크기로, 개인 쪽에서는 달성 여부로 다르게 해석된다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 확인 완료된 제출들의 볼륨 누적. 볼륨은 제출 한 건의 무게에 횟수를 곱한 값이고, 개인별 누적은 각 팀원이 그 종목에서 기간 내내 쌓은 확인 완료 볼륨의 합이며, 팀별 누적은 소속 팀원들의 개인별 누적을 더한 값이다.

```json
{
  "개인별_누적": { "참가자": "확인 완료 제출의 볼륨 누적 합" },
  "팀별_누적": { "팀": "소속 팀원 볼륨 누적 합의 단순 합" }
}
```

**출력 데이터:** 실시간 팀 순위, 팀 내 기여도 순위, 개인 목표 달성 현황.

```json
{
  "팀_순위": "팀별 볼륨 누적 합 기준 전체 팀 순위",
  "팀_내_기여도_순위": {
    "팀": [
      {
        "팀원": "팀원 식별",
        "기여도_비율": "소속 팀 누적 합 중 자기 볼륨 비율",
        "순위": "팀 안에서의 기여도 순위. 비율이 같은 팀원은 공동 순위"
      }
    ]
  },
  "개인_목표_달성_현황": {
    "참가자": {
      "달성_여부": "달성 또는 미달(순위 아님)",
      "진행_정도": "목표까지의 진행 정도"
    }
  }
}
```

#### 3. 기능 요구사항

- 팀 점수는 팀원 볼륨 누적 합의 단순 합이다. 편성 단계에서 팀 인원을 같게 맞췄으므로 머릿수 차이로 생기는 유불리가 없어 단순 합이 그대로 공정한 비교가 된다.
- 팀 순위는 팀별 볼륨 누적 합으로 전체 팀을 세운다. 팀 점수가 동점이어서 순위가 갈리면 그 점수에 먼저 도달한 팀을 위에 둔다.
- 팀 내 기여도 순위는 한 팀 안에서 팀원을 그 팀 누적 합 중 자기 볼륨 비율로 세운다. 이 순위가 팀 상금 분배의 근거이며, 대상은 그 팀 팀원으로 한정된다. 기여도 비율이 같은 팀원은 공동 순위로 처리한다.
- 개인 목표 달성 현황은 순위가 아니라 달성·미달 판정과 목표까지의 진행 정도다. 전체나 팀에서 개인을 볼륨으로 줄 세우지 않는다. 전체 참가자를 볼륨으로 통틀어 세우는 개인 순위는 두지 않는다.
- 참가자가 중도 이탈하면 그 사람이 쌓은 볼륨을 팀 점수에서 뺀다. 팀 점수는 항상 현재 활동 중인 팀원의 수행만 반영한다.
- 인바디나 부위 둘레 같은 신체 결과 지표와 운동 시간은 순위 산정에 쓰지 않는다. 이 서비스는 운동의 효과가 아니라 수행 여부와 수행의 크기를 다룬다.

**접근 패턴 (설계 단계 구현 근거)**

팀 순위는 확인 한 건이 확정될 때 그 제출자가 속한 팀의 볼륨 누적 합에 이번 건의 볼륨을 더하고, 전체 팀을 볼륨 누적 합 내림차순으로 정렬한 상태를 반영한다. 중도 이탈이 발생하면 그 팀원의 볼륨을 팀 누적 합에서 빼고 정렬에 반영한다. 조회 시 전체 팀 순서와 자기 팀 순위, 상금 경계 기준 자기 팀의 안팎을 반환한다. 팀 점수 동점은 위 규칙대로 그 점수에 먼저 도달한 팀을 위에 둔다.

팀 내 순위는 같은 확인 사건이 그 제출자 개인의 볼륨 누적에도 이번 건의 볼륨을 더하고, 그 팀 안에서 팀원을 개인 볼륨 누적 기준으로 정렬한 상태를 반영한다. 각 팀원의 팀 누적 합 대비 비율을 산출하며, 조회 시 팀 내 팀원 순위와 기여도 비율을 반환하고 비율이 같은 팀원은 공동 순위로 묶는다.

위 갱신은 단일 처리 흐름 기준으로 기술한다. 여러 확인이 동시에 들어올 때의 동시성 처리는 이 단계에서 다루지 않는다. 이 접근 패턴을 만족하는 자료구조와 구현체 선택, 동시성 기법, 최적화는 이 단계에서 확정하지 않고 기능이 온전히 동작하는 수준으로 둔다. 구현 이후 계측을 근거로 다시 연다. Redis와 분산 인스턴스는 이 단계 범위 밖이다. (미확정)

### F008 - 마감과 확인 윈도우

#### 1. 기능 개요

수행 마감 시각 이후에도 소급 등록 유예 24시간 동안은 수행 기간 안의 날짜를 지정한 늦은 제출을 받고, 그 유예가 끝나는 시점에 제출이 완전히 마감된다. 마감 직전이나 유예 구간에 올라온 제출이 확인될 시간을 주기 위해, 제출 완전 마감 뒤에 확인 윈도우를 챌린지 전체에 하나 연다. 윈도우 길이는 F001 등록 시 챌린지마다 정한 값이며, 제출 완전 마감 시각에 이 길이를 더한 시점이 윈도우 종료다. 확인 윈도우가 열리는 시점에는 이미 모든 제출이 끝나 있어 확인만 남는다. 윈도우가 끝나면 그때까지 확인되지 않은 제출을 만료로 처리하고, 윈도우 종료 시점에 정산이 실행된다.

수행 마감부터 정산까지의 시각 관계는 다음과 같다.

```
수행 마감 ──[소급 등록 유예 24시간]──> 제출 완전 마감 ──[확인 윈도우 길이]──> 윈도우 종료·정산
```

#### 2. 사용자 시나리오

수행 마감 뒤 소급 유예를 거쳐 제출이 완전히 닫히고, 이어서 확인 윈도우가 열리며, 윈도우가 끝나면 만료 처리와 정산이 실행된다.

**2.1. 수행 마감과 소급 유예**
수행 기간의 마감 시각 이후 새 수행은 없다. 다만 마감 직전에 수행한 운동을 미처 못 올린 참가자를 위해, 마감 뒤 24시간 동안은 수행 기간 안의 날짜를 지정한 소급 제출을 받는다(소급 등록 유예는 F005 참조). 이 24시간이 지나면 제출이 완전히 마감되고, 이후 어떤 제출도 받지 않는다.

**2.2. 확인 윈도우**
제출 완전 마감 뒤 확인 윈도우가 챌린지 전체에 하나 열린다. 윈도우 길이는 F001에서 그 챌린지에 정한 값이다. 팀원 확인은 수행 기간과 소급 유예 구간에도 계속 받으며, 이 윈도우는 제출이 전부 끝난 뒤 아직 확인되지 않은 제출의 확인만 받는 마지막 구간이다. 윈도우가 끝난 뒤 들어오는 확인은 종료 시각과 비교해 거부한다.

**2.3. 만료 처리와 정산 실행**
윈도우가 끝나면 그때까지 확인되지 않은 제출을 만료로 처리하고, 윈도우 종료 시점에 정산(F009)을 실행한다.

#### 3. 기능 요구사항

- 수행 마감 시각에 소급 등록 유예 24시간을 더한 시점이 제출 완전 마감이다. 유예 구간에는 수행 기간 안의 날짜를 지정한 소급 제출만 받고, 제출 완전 마감 이후에는 어떤 제출도 받지 않는다.
- 제출 완전 마감 뒤 확인 윈도우는 챌린지마다 하나만 둔다. 윈도우 길이는 F001 등록값이며, 제출 완전 마감 시각에 그 길이를 더한 시점이 윈도우 종료다.
- 윈도우 종료 시점까지 확인되지 않은 제출은 만료로 처리한다.
- 윈도우 종료 뒤 들어오는 확인은 종료 시각과 비교해 거부한다.
- 윈도우 종료 시점이 정산의 실행 시점이다.

### F009 - 정산

#### 1. 기능 개요

확인 윈도우가 끝나면 정산이 한 번 실행된다. 정산은 개인 판정부터 팀 내 분배까지 순서가 있는 네 단계로 이뤄지고, 앞 단계의 결과가 뒤 단계의 입력이 된다. 개인의 미달이 곧 팀 상금의 재원이므로 개인 정산과 팀 정산이 차감분으로 연결된다. 정산은 이 전체가 부분적으로 반영되어서는 안 되고 전부 반영되거나 전부 반영되지 않아야 하며, 재원 등식이 맞아야 끝난다. 중복 실행은 이중 지급을 만들므로 정산은 단 한 번만 실행된다. 이 네 단계 정산은 챌린지가 정상 종료된 경우다. 모집 미달로 무산된 경우는 이 단계를 타지 않고 예치금을 전액 반환하며, 무산 판정과 반환은 F002가 다룬다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 윈도우 종료 시점의 정산 입력.

```json
{
  "개인별_목표_달성_여부": { "참가자": "달성 또는 미달" },
  "팀별_볼륨_누적_합": { "팀": "팀원 볼륨 누적 합" },
  "팀원별_볼륨_비율": { "팀": { "팀원": "팀 누적 합 중 비율" } },
  "기본_상금풀": "운영이 기본으로 넣는 상금풀",
  "상금_받을_팀_수": "전체 팀 수보다 작은 값(등록 시 정함)"
}
```

**출력 데이터:** 개인 환급·차감 결과와 팀 상금 분배 결과.

```json
{
  "개인_정산": { "참가자": "환급 또는 차감 금액" },
  "팀_상금_분배": { "팀": { "팀_몫": "순위 차등 분배액", "팀원별": { "팀원": "기여도 차등 분배액" } } },
  "시스템_환수분": "순위 분배와 팀 내 분배의 나눗셈에서 남은 나머지 금액의 합"
}
```

#### 3. 기능 요구사항

**정산 단계**

정산은 앞 단계 결과가 뒤 단계 입력이 되는 네 단계 순서로 실행된다.

1. 개인 판정. 참가자마다 목표 달성 여부로 환급 대상과 차감 대상을 가른다. 미달자 차감분의 총합이 이 단계에서 나오고, 이 총합이 팀 상금풀의 한 재원이 된다.
2. 상금풀 확정. 기본 상금풀에 1단계 차감분 총합을 더해 팀 상금풀을 확정한다. 개인 정산 결과가 있어야 상금풀이 정해지므로 이 단계는 1단계 뒤에 온다.
3. 팀 순위 분배. 확정된 상금풀을 팀 순위로 상금 받을 팀에 차등으로 가른다. 순위 경계 밖 팀은 받지 않으며, 이 단계에서 각 수령 팀의 몫이 정해진다.
4. 팀 내 분배. 각 팀 몫을 그 팀 안에서 팀원 볼륨 비율로 나눈다. 팀원 개인이 받을 금액이 이 단계에서 정해진다.

**판정과 분배 규칙**

- 개인 보상은 목표 달성과 미달 둘 중 하나로 가르는 판정이다. 개인 목표는 시스템이 개인 목표 기준값에 참가자 계수를 곱해 잡은 개인 강도 하한과, 참가자가 정한 목표 빈도 두 조건으로 이뤄진다. 강도 하한 이상인 확인 완료 제출을 수행 기간 전체 누적으로 세어, 목표 빈도에서 도출한 기간 전체 목표 횟수를 채우면 달성으로 보고 예치금을 환급하고, 못 채우면 미달로 보고 차감한다(기간 누적 충족). 기간 안 어느 날짜에 채웠는지는 따지지 않는다. 기간 전체 목표 횟수는 완전 주기만 세는 내림 규칙으로 도출한다. 주 며칠(간격 k)이면 내림(D ÷ 7) × k, 며칠에 한 번(간격 n)이면 내림(D ÷ n)이며, D는 수행 기간이 걸치는 달력 날짜 수다. 도출 값이 0이면 1로 하고, 완전 주기 밖 끝자락 날짜의 제출도 누적 집계에는 포함된다. 강도 하한을 넘긴 초과분은 환급에 더 얹지 않는다.
- 상금풀은 운영이 넣는 기본 상금풀과 미달자 차감분 두 재원으로 채운다. 기본 상금풀이 있으므로 미달자가 없는 챌린지에서도 상금풀이 0이 되지 않는다.
- 팀 순위 분배는 1등이 가장 많이 받고 등수가 내려갈수록 적게 받으며, 순위 경계 밖의 팀은 받지 않는다. 팀 점수가 동점이어서 순위가 갈리면 그 점수에 먼저 도달한 팀을 위에 둔다. 등수별 몫은 등차 가중으로 정한다. 상금 받을 팀 수를 P라 할 때 순위 i 팀의 가중치는 P − i + 1이고, 팀 몫은 상금풀 × 가중치 ÷ 가중치 합(P × (P + 1) ÷ 2)의 내림이다. 내림에서 남는 나머지는 시스템 환수 규칙을 그대로 따른다.
- 팀 내 분배에서 팀원의 볼륨 비율에 상한을 두지 않는다. 한 팀원이 볼륨을 압도적으로 쌓아 상금을 거의 독점하더라도 그대로 둔다. 많이 든 만큼 많이 받는 것이 절대 수행량 경쟁의 원칙이기 때문이다.
- 순위 분배와 팀 내 분배의 나눗셈에서 남는 나머지 금액은 시스템이 환수한다. 나머지를 참가자에게 다시 배분하지 않으며, 환수분은 버려지지 않고 재원 등식의 한 항으로 잡힌다.

**정합성과 실행**

- 정산 전체는 부분적으로 반영되어서는 안 되고, 전부 반영되거나 전부 반영되지 않아야 한다. 이를 어떤 방식으로 보장할지는 설계 단계에서 정한다.
- 같은 차감분을 개인에서 빼는 값과 팀 상금풀에 더하는 값이 어긋나지 않아야 한다. 환급 합은 달성자 환급 재원과 맞아야 하고, 팀원에게 분배된 상금 합과 시스템 환수분의 합은 상금풀과 정확히 맞아야 정산이 끝난다.
- 정산은 윈도우 종료 시점에 단 한 번만 실행한다. 중복 실행을 막아 이중 지급을 차단한다.

### F010 - 예치 충전

#### 1. 기능 개요

참가자가 금액을 입력하고 확인하면 본인 예치 잔액에 반영된다. 외부 결제나 계좌 연동 없이 잔액 숫자만 움직이는 모의 예치다. 예치금은 챌린지 참가 신청 시 차감되고, 정산 시점에 환급·차감·무산 반환으로 갈린다.

#### 2. 입력 및 출력 데이터 모델

**입력 데이터:** 참가자가 예치·충전 화면에서 입력한다.

```json
{
  "충전_금액": "참가자가 입력한 충전 금액"
}
```

**출력 데이터:** 갱신된 예치 잔액과 챌린지별 예치 상태.

```json
{
  "예치_잔액": "충전이 반영된 본인 예치 잔액",
  "챌린지별_예치_상태": { "챌린지": "진행 중 / 환급 / 차감 / 무산 반환 중 현재 상태" }
}
```

#### 3. 기능 요구사항

- 충전은 외부 결제나 계좌 연동 없이 금액을 입력하고 확인하면 잔액에 반영되는 모의 방식으로 둔다.
- 충전 금액은 0보다 큰 값이어야 한다. 음수나 0은 받지 않는다.
- 참가 신청(F002) 시 예치금을 잔액에서 차감한다.
- 챌린지별로 어디에 얼마를 예치했고 그 예치금이 지금 어떤 상태(진행 중, 환급, 차감, 무산 반환)인지 추적한다.

### 기능별 상세 요구사항

**F001: 챌린지 등록과 성립 검증**
- 입력 조건: 닫힌 목록에서 고른 단일 카테고리와 그 안의 단일 종목, 팀 정원, 팀 수, 상금 받을 팀 수, 참가 예치금, 날짜·시각으로 잡은 모집·수행 기간, 확인 윈도우 길이, 팀 볼륨 집계용 인증 빈도(하루 상한·주기)
- 처리 요구사항: 목표 인원을 정원×팀수로 산출, 상금 받을 팀 수 < 팀 수 검사, 하루 인증 횟수 상한 존재 검사, 참가 예치금 0보다 큰 값 검사
- 출력 요구사항: 제약을 통과한 챌린지를 모집 상태로 공개
- 에러 처리: 상금 받을 팀 수가 팀 수 이상이거나 하루 상한이 없으면 등록 거부

**F002: 챌린지 참가 신청**
- 입력 조건: 개인 목표(강도 계수, 목표 빈도)를 신청과 함께 받으며, 이 값이 있어야 신청 성립
- 개인 목표 처리: 강도 계수 1 이상(미입력 시 1), 강도 하한은 개인 목표 기준값(F003)에 계수를 곱해 시스템이 산출, 목표 빈도는 하루 상한 안으로 제한
- 성공 기준: 목표 인원 도달 시 편성 실행과 챌린지 시작
- 실패 처리: 모집 기간 종료까지 미달이면 무산하고 예치금 전액 반환

**F003: 기준 측정 인증**
- 판단 기준: 신청 시점 기준 그 종목 최근 30일 확인 완료 기록 유무로 측정 요구 여부를 가른다
- 요청 시점: 윈도우 내 기록 없는 참가자에게 F002 참가 신청 절차 안에서 기준 측정을 받음(과거 측정이 있어도 윈도우를 벗어나면 재측정)
- 분기 조건: 윈도우 내 기록 있으면 그 확인 완료 제출의 볼륨 단순 평균 사용, 없으면 기준 측정 1회 값 사용
- 검증: 팀원 확인 없이 F005와 동일한 기계 검증(해시 중복, 메타데이터 정합성)만 적용
- 산출 용도: 편성 실력과 개인 목표 기준값은 동일 값(최근 30일 볼륨 1회 평균). 앞의 것은 팀 편성 입력, 뒤의 것은 개인 강도 하한 기준
- 윈도우: 길이 30일은 잠정 설정값이며 운영 데이터 축적 후 재검토
- 한계: 낮게 인증하는 조작은 이 검증 모델로 완전히 막지 못하며 구조적 한계로 수용

**F004: 팀 편성**
- 실행 조건: 외부 호출이 아니라 F002 목표 인원 충족 시 실행되는 내부 연산
- 동시성: 인원 충족 판정과 편성 시작을 단일 실행으로 묶어 정원 초과와 이중 실행 방지
- 균형 축: 실력 합 균형은 제약(합 편차 ÷ 전체 팀 합의 평균 ≤ r%, 허용폭으로 판정), 실력 분포 균형은 제약 안에서 최소화하는 목표(팀별 분산의 최대·최소 차). 두 편차를 종합 점수로 합치지 않음
- 로직 플로우: 준비(허용폭 산출), 시작 배정 2개 생성(교대 방향 배정·내림차순 그리디), 합 필터와 최선 수락의 교환 반복(개선 후보 소진 시 종료), 채택과 시작 상태 전이
- 입력 신호: 자기 신고 등급이 아니라 최근 30일 확인 완료 제출 볼륨의 1회 평균(편성 실력) 또는 기준 측정 1회 볼륨
- 보장: 모든 팀 인원 동일, 나머지 인원 없음, 편성 무실패, 반드시 종료, 결정성, 편성 후 챌린지 시작 상태로 전이
- 미확정: 합 상한 r은 잠정 설정값(절충 곡선 꺾임점으로 결정). 증명된 후보 생성 생략·증분 재계산·병렬 분할 같은 구현 최적화는 구현 이후 계측으로 결정

**F005: 인증 제출과 시스템 검증**
- 처리 순서: 제출 접수 후 시스템 기계 검증을 거쳐 확인 대기로 진입
- 검증 항목: 해시 중복, 메타데이터 정합성, 시간 윈도우(하루 상한·주기)
- 날짜 연결: 등록 시점을 기록하고 제출을 참가자가 지정한 수행 날짜에 연결해 저장(수행 날짜는 제출 시점 기준 24시간 안, 소급 등록 유예), 날짜 안 시간대는 구분하지 않음. 하루 상한·주기 판정도 수행 날짜 기준
- 조회 요구: 날짜에 연결된 인증 내역을 날짜 단위로 조회 가능(열람 화면은 내 제출 현황 쪽)
- 경계: 사진 내용 판별은 다루지 않음
- 실패 처리: 검증 실패 시 거부하고 사유 반환

**F006: 팀원 확인과 점수 반영**
- 사용자 권한: 같은 팀의 팀원, 본인 자기 제출 확인 차단
- 확인 단위: 확인 대기 제출 한 건 단위의 일회성 전이, 이미 확인 완료된 제출은 재확인하지 않음
- 정족수: 확인 완료에 필요한 팀원 확인 수 1
- 반영: 확인 시 그 한 건의 볼륨을 개인·팀 누적에 증분 가산, 개인 기록 전체 재계산 없음, 중복 확인에도 이중 가산 방지
- 상태: 확인 대기가 진입 상태, 확인 완료·반려·만료는 종착. 반려는 무효 종료이며 재제출은 새 제출 건으로 확인 대기부터 시작

**F007: 순위와 개인 목표 현황 산출**
- 산정 기준: 팀 순위는 팀별 볼륨 누적 합, 팀 내 순위는 팀원 볼륨 비율, 개인은 목표 달성·미달 판정
- 동점 처리: 팀 순위는 먼저 도달한 팀 우위, 팀 내 순위는 기여도 동점 시 공동 순위
- 경계: 개인은 순위가 아니라 달성 현황이며, 전체 참가자를 볼륨으로 세우는 개인 순위는 없음
- 이탈 반영: 중도 이탈자의 볼륨을 팀 점수에서 차감
- 미확정: 정렬 유지·순위 조회를 받칠 자료구조는 계측 후 설계 단계에서 확정

**F008: 마감과 확인 윈도우**
- 처리 순서: 수행 마감 후 소급 유예 24시간 동안 소급 제출 수용, 유예 종료 시 제출 완전 마감, 단일 확인 윈도우, 윈도우 종료 시 만료 처리와 정산 실행
- 윈도우 길이: F001 등록값, 제출 완전 마감 시각(수행 마감 + 24시간)에 그 길이를 더한 시점이 윈도우 종료
- 타임아웃: 윈도우 종료 뒤 들어오는 확인은 종료 시각과 비교해 거부
- 단일성: 확인 윈도우는 챌린지마다 하나

**F009: 정산**
- 처리 순서: 개인 판정과 차감분 집계, 기본 상금풀에 차감분 더해 상금풀 확정, 팀 순위 분배, 팀 내 기여도 분배의 네 단계(앞 단계 결과가 뒤 단계 입력)
- 나머지 처리: 분배 나눗셈에서 남은 금액은 시스템 환수, 재원 등식의 한 항으로 잡음
- 정합성: 정산 전체가 부분 반영 없이 전부 반영되거나 전부 안 됨, 차감분 정합, 분배 합과 환수분의 합이 상금풀과 일치(보장 방식은 설계 단계)
- 트랜잭션: 윈도우 종료 시점에 단 한 번만 실행, 중복 실행 차단으로 이중 지급 방지
- 분배 규칙: 팀 사이 순위 차등(등차 가중, 순위 경계 밖 0), 팀 안 기여도 비율 차등(상한 없음)
- 무산 분기: 모집 미달 무산은 이 네 단계를 타지 않고 예치금 전액 반환(판정·반환은 F002)

**F010: 예치 충전**
- 외부 연동: 외부 결제·계좌 연동 없는 모의 충전
- 처리: 금액 입력·확인 시 잔액 반영
- 입력 검증: 충전 금액은 0보다 큰 값만 허용, 음수·0 거부
- 추적: 챌린지별 예치 상태(진행 중, 환급, 차감, 무산 반환) 관리

## 4. 비기능 요구사항

이 절은 기획서와 프로젝트 방향성에서 도출되는 품질 속성만 적는다. 정량 목표와 정책 세부는 근거가 없는 한 확정하지 않고 미확정으로 표기한다.

### 4.1 정합성과 동시성

- **정산 정합성:** 같은 차감분을 개인 예치금에서 빼는 동작과 팀 상금풀에 더하는 동작이 어긋나지 않아야 한다. 환급 합과 상금 합이 각 재원과 정확히 맞아야 정산을 완료로 본다.
- **단일 실행 보장:** 정산은 윈도우 종료 시점에 단 한 번만 실행되어야 한다. 중복 실행이 이중 지급을 만들지 않도록 단일 실행을 보장한다.
- **확인 한 건의 두 집계 반영:** 팀원 확인 한 건이 확정되면 그 제출의 볼륨이 제출자 개인 누적과 소속 팀 누적 두 곳에 함께 더해진다. 한 건이 두 집계에 반영되는 처리이며, 여러 확인이 동시에 들어올 때의 동시성 처리는 이 단계에서 다루지 않는다.
- **상태 전이 무결성:** 제출은 확인 대기, 확인 완료, 반려, 만료 사이를 정해진 전이로만 이동해야 한다.
- **모집·잔액 동시 접근:** 모집 인원 도달 판정과 예치 잔액 차감처럼 동시 접근이 일어나는 지점에서 정합성을 유지해야 한다.

### 4.2 계산 성능

- **팀 편성 계산:** 목표 인원이 찬 시점에 참가자 전체를 한꺼번에 보고, 실력 합 균형을 제약으로 실력 분포 균형을 목표로 두는 교환 반복으로 배정을 탐색한다. 반복 1회의 평가 후보 수가 팀 수와 인원에 따라 커지는 CPU 바운드 구간이며, 구현 최적화와 병렬화 모델은 부하 계측 뒤에 정한다.

### 4.3 데이터 무결성

- **인증 무결성:** 시스템 검증이 해시 중복으로 같은 사진 재사용을 차단하고, 메타데이터 정합성을 검사하며, 시간 윈도우로 인증 빈도를 강제한다.
- **확인 무결성:** 본인이 자기 제출을 확인하는 것을 차단한다.

### 4.4 미확정 영역

다음 항목은 기획서에 근거가 없어 이 단계에서 확정하지 않는다. 추정으로 채우지 않는다.

- 응답 시간, 동시 처리량, 가용률 같은 정량 성능 목표
- 사용자 인증 방식, 권한 체계, 데이터 암호화, 감사 로깅 정책
- 접근성 기준, 다국어 지원, 브라우저 지원 범위, 반응형 대응 범위
- 입출력 데이터 모델의 구체 타입, 식별자 형식, 시각 표현 형식

### 기능 간 의존성

```
F010 (예치 충전)
  └─> F002 (참가 신청)에 잔액 차감으로 연결

F001 (챌린지 등록)
  └─> F002 (참가 신청)
        ├─> F003 (기준 측정 인증)   [최근 30일 내 그 종목 기록 없는 참가자에 한해, 팀원 확인 없이 기계 검증만]
        └─> F004 (팀 편성)
              └─> F005 (인증 제출과 시스템 검증)
                    └─> F006 (팀원 확인과 점수 반영)
                          ├─> F007 (순위·현황 산출)
                          └─> F008 (마감과 확인 윈도우)
                                └─> F009 (정산)
                                      └─> F010 (예치 상태)에 환급·차감 반영
```

### 용어 정의

- **인스턴스:** 단일 주제와 유한 기간을 가지는 개별 챌린지.
- **제출 상태:** 확인 대기, 확인 완료, 반려, 만료. 확인 대기는 팀원 확인을 기다리는 진입 상태이고, 확인 완료(확인되어 볼륨이 반영된 상태), 반려(반려로 무효 종료된 상태), 만료(윈도우 종료까지 확인되지 않은 상태)는 모두 종착이다. 종착에 도달한 제출은 다른 상태로 이동하지 않으며, 반려 후 다시 올린 제출은 별개의 새 제출 건이다.
- **수행 날짜:** 참가자가 제출할 때 지정하는, 운동을 실제로 수행한 날짜. 제출이 이 날짜에 연결되어 저장되며 하루 상한과 인증 주기 판정의 기준이다.
- **소급 등록 유예:** 수행 시점으로부터 24시간 안에는 지난 날짜를 수행 날짜로 지정해 제출할 수 있는 유예. 하루치 지각 등록은 받아주되 여러 날을 몰아서 소급하는 것은 막는다.
- **제출 완전 마감:** 수행 마감 시각에 소급 등록 유예 24시간을 더한 시점. 이 시점 이후에는 어떤 제출도 받지 않는다.
- **확인 윈도우:** 제출 완전 마감 이후 늦은 확인을 수용하는 챌린지 전체 단일 기간. 이 기간이 열리는 시점에는 모든 제출이 끝나 있어 확인만 남는다.
- **편성 실력:** 팀 편성(F004) 입력으로 쓰는 값. 신청 시점 기준 그 종목 최근 30일 확인 완료 제출 볼륨의 1회 평균. 윈도우 내 기록이 없으면 기준 측정 1회 볼륨. 한 번 수행의 수준을 근사하며, 신규 참가자의 기준 측정값과 같은 종류의 숫자로 비교된다.
- **개인 목표 기준값:** 개인 목표 강도 하한을 정하는 기준. 편성 실력과 동일한 값이며 쓰이는 자리만 다르다. 윈도우 내 기록이 없으면 기준 측정 1회 볼륨. 한 번 수행의 무게 수준을 근사한다.
- **개인 목표:** 참가자가 참가 신청 때 정하는 자기 보상 목표. 시스템이 개인 목표 기준값에 참가자 계수(1 이상)를 곱해 잡는 강도 하한과, 참가자가 정하는 목표 빈도로 이뤄진다. 강도는 참가자가 신고하지 않고 시스템이 산출한다.
- **달성률:** 목표 대비 수행 정도. 개인 보상 환급의 판정 기준이며, 강도 하한 이상인 확인 완료 제출을 기간 누적으로 세어 기간 전체 목표 횟수를 채웠는지를 달성과 미달 둘로 판정한다.
- **볼륨:** 제출 한 건의 무게에 횟수를 곱한 값. 팀 점수와 기여도 산정에 쓴다.
- **정족수:** 제출이 확인 완료로 넘어가는 데 필요한 팀원 확인 수. 기본값 1.

### 추가 결정 필요

기획 단계에서 확정하지 않았고 설계 단계에서 자르는 항목이다.

- 팀 편성의 합 상한 r 값(잠정 설정값, 참가자 구성 모양별 절충 곡선의 꺾임점으로 결정)과 증명된 후보 생성 생략·증분 재계산·병렬 분할 같은 구현 최적화. 편성 절차 자체는 F004에서 종결한다
- 순위 산출의 자료구조와 구현체 선택, 동시성 처리, 최적화. 요구사항 단계에서는 접근 패턴만 세우고, 구현 이후 계측을 근거로 결정한다. Redis와 분산 인스턴스는 범위 밖이다
