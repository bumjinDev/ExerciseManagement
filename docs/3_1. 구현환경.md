## 2. 인증 방식 요약 (확인된 동작)

Wherehouse의 인증은 세션 기반이 아니라 JWT 기반 무상태 방식이다. 모든 SecurityFilterChain이 `SessionCreationPolicy.STATELESS`로 설정되어 있다.

- 로그인: `/login` 경로에 `LoginFilter`(`UsernamePasswordAuthenticationFilter` 상속)가 등록되어 있다. JSON(`application/json`)과 폼(`application/x-www-form-urlencoded`) 두 형식의 아이디·비밀번호 요청을 모두 받는다.
- 인증 수행: `UserAuthenticationProvider`가 `UserEntityRepository`로 사용자를 조회하고 `BCryptPasswordEncoder.matches()`로 비밀번호를 검증한다. 회원 조회 경로는 `UserEntityDetailService`(`UserDetailsService` 구현)다.
- 토큰 발급: 인증 성공 시 HMAC-SHA256 서명 JWT를 생성하고(클레임: `username`, `userId`, `roles`) `Authorization`이라는 HttpOnly 쿠키에 담아 반환한다.
- 요청 검증: 보호 경로에는 `JwtAuthProcessorFilter`가 쿠키의 JWT를 검증한다. 인증 실패는 401(EntryPoint), 인가 실패는 403(AccessDeniedHandler)으로 응답한다.
- 로그아웃: `/logout`에서 `CookieLogoutHandler`가 쿠키를 제거한다.
- 패스워드 인코더: `BCryptPasswordEncoder` 빈. 출처: `src/main/java/com/wherehouse/JWT/SecurityConfig/SecurityConfig.java`.

### 3.2 회원 테이블 구조 (엔티티 매핑 기준)

`membertbl` (엔티티 `MembersEntity`):

| 컬럼 | 설명 |
|------|------|
| id | 아이디, PK |
| pw | 비밀번호 (BCrypt 해시 저장) |
| nickName | 닉네임 |
| tel | 전화번호 |
| email | 이메일 |
| joinDate | 가입 날짜 (`java.sql.Date`) |

`userentity` (엔티티 `AuthenticationEntity`, 인증 전용):

| 컬럼 | 설명 |
|------|------|
| userid | 아이디, PK, not null |
| username | 사용자 이름(닉네임), not null |
| password | 비밀번호 해시, not null |

`userentity_roles` (`@ElementCollection`): `userid`(FK) + `roles` 문자열 목록. 가입 시 기본 `ROLE_USER` 하나가 부여된다.

회원 테이블과 인증 테이블이 이원화되어 있고, 가입·수정 시 `MemberService`가 두 테이블을 함께 갱신한다. 가입 로직에는 예약어 아이디·닉네임 차단(`admin`, `root` 등 10종), 아이디·닉네임 중복 검사, BCrypt 인코딩이 포함된다.

### 3.3 이식 시 조정이 필요한 지점

- Redis 의존 분리: `MemberService`와 `LoginFilter`가 `com.wherehouse.redis.handler.RedisHandler`를 참조한다(생성자 주입, import). 새 프로젝트의 순위 누적 저장 전략과 캐시는 계측 후 결정하는 미확정 항목이므로(설계 명세서 8.3.3, 10.3), 이 참조를 제거하거나 대체한 뒤 이식한다.
- JWT 만료 시간: `JWTUtil.EXPIRATION_TIME`이 부하 측정을 위해 24시간으로 임시 연장된 상태다(코드 주석에 원래 값 1시간 명시). 이식 시 이 프로젝트의 정책으로 확정해서 설정한다.
- 로그인 성공 리다이렉트: `LoginFilter`가 `/wherehouse/members/loginSuccess`로 리다이렉트한다. 새 프로젝트의 경로와 응답 방식(페이지 이동 또는 JSON 응답)에 맞게 바꾼다.
- 쿠키 보안 속성: JWT 쿠키가 `setSecure(false)` 상태다. 배포 환경(HTTPS 여부)에 맞춰 조정한다.
- 권한 체계: Wherehouse는 `ROLE_USER` 단일 역할과 경로별 `authenticated`/`permitAll` 구성을 쓴다. 이 부분은 차용하지 않는다(아래 4장).
- 테이블 생성: 두 테이블과 역할 테이블을 새 프로젝트 전용 스키마에 새로 만든다. `ddl-auto: none`이므로 DDL은 직접 작성한다.
- 컨텍스트 경로·포트: Wherehouse는 `context-path /wherehouse`, 포트 8185를 쓴다. 새 프로젝트 값은 따로 정한다.
- 환경 변수: `JWT_SECRET_KEY`(Base64 인코딩된 HMAC 키)를 실행 환경에 준비해야 한다.

## 4. 이 프로젝트에서 새로 만드는 것

- 전용 DBMS 계정과 스키마: Wherehouse의 계정·스키마를 공유하지 않는다. 회원 데이터를 포함한 모든 테이블을 이 프로젝트 전용 계정과 스키마에 분리해 운용한다(설계 명세서 3.1, 11.1).
- 권한 체계: 두 프로젝트의 비즈니스 로직이 달라 Wherehouse의 권한 구성을 차용하지 않는다. 이 프로젝트의 도메인(챌린지 참가자, 팀원 확인 권한 등)에 맞춰 따로 설계하며 세부는 구현 단계에서 확정한다.

## 5. 환경 전제

- 개발은 로컬 환경에서 시작한다.
- 테스트가 끝나면 미니 PC 서버로 이관한다. Wherehouse의 설정 파일에도 로컬용과 미니 PC용 접속 항목이 주석으로 전환되는 구조가 있으므로, 새 프로젝트는 프로파일 분리(local/prod 등)로 같은 전환을 처리하는 것을 권장한다.
