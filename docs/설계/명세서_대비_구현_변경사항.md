# 명세서 대비 구현 변경사항 (누적 기록)

최초 작성: 2026-07-07
규칙: 요구사항·설계 명세서와 다르게 구현되거나 새로 추가된 모든 지점을 이 파일에 누적 기록한다.
각 항목에 [유형]과 사유를 남긴다. 유형: 추가(명세에 없던 것) / 변경(명세와 다르게) / 확정(명세가 "구현 단계 확정"으로 미룬 값의 확정).

## 4. 비즈니스 로직 해석·확정 (단계 3)

명세가 정의하지 않았거나 해석 여지가 있어 기준 구현에서 다음과 같이 확정했다.

| # | 유형 | 내용 |
|---|------|------|
| 4-1 | 확정 | 기본 상금풀: 등록 시 시스템 프로퍼티 app.base-prize-pool(잠정 500,000원) 값을 저장 |
| 4-2 | 확정 | 팀 편성 합 상한 r: 프로퍼티 app.formation-sum-cap-percent(잠정 10%). 계측 후 재확정 TODO |
| 4-3 | 확정 | 카테고리·종목 닫힌 목록(시작값): 가슴[벤치프레스, 인클라인 벤치프레스, 체스트프레스], 등[데드리프트, 바벨로우, 랫풀다운], 어깨[오버헤드프레스, 시티드 숄더프레스], 팔[바벨컬, 라잉 트라이셉스 익스텐션], 하체[스쿼트, 레그프레스]. application.yaml에서 관리 |
| 4-4 | 확정 | 메타데이터 정합성(E-SUB-META-MISMATCH) 기준 구현: ① 파일이 파싱 가능한 이미지인지 ② EXIF 촬영 시각이 있으면 수행 날짜(linked_date)와 ±24시간 정합인지. EXIF 없으면 통과(스크린샷 등 고려). 라이브러리 metadata-extractor 사용 |
| 4-5 | 확정 | 인증 주기 판정: '며칠에 한 번'(간격 n)=마지막 인정 인증 날짜와 새 수행 날짜의 차이 ≥ n일. '주 며칠'(간격 k)=그 수행 날짜가 속한 달력 주(월요일 시작)의 인증 날짜 수 < k. 인정 인증 = 반려·만료 아닌(PENDING·CONFIRMED) 일반 제출 |
| 4-6 | 확정 | 하루 상한 판정: 같은 수행 날짜의 PENDING·CONFIRMED 일반 제출 수 < daily_cap |
| 4-7 | 확정 | 팀 순위 동점("먼저 도달한 팀 우위"): 반영된 확인의 최종 confirmed_at이 빠른 팀이 위. 확인이 없는 0점 동점은 팀 ID 순 |
| 4-8 | 확정 | 이탈(WITHDRAWN) 참가자: 팀 누적·팀 내 기여도 목록·순위 조회 대상에서 제외 |
| 4-9 | 확정 | 정산 원자성(명세 8.4.2 "방법은 구현 확정"): 단일 DB 트랜잭션(@Transactional)으로 4단계 전체 처리, 검산 불일치 시 예외로 전체 롤백 |
| 4-10 | 확정 | 원장 회계 규칙: 원장 amount 합 = 현재 잔액. CHARGE(+), JOIN_DEBIT(−), REFUND(+), VOID_RETURN(+). FORFEIT(차감 확정)는 amount 0의 상태 기록 — 금액 이동은 신청 시 JOIN_DEBIT에서 이미 발생했고, 몰수액은 정산 검산에서 participation.deposit_amount로 계산 |
| 4-11 | 확정 | 상금(TEAM_PRIZE/MEMBER_PRIZE)은 예치 잔액에 넣지 않고 분배 기록으로만 남김 — 명세에 상금 지급용 원장 항목이 없음(모의 서비스 범위) |
| 4-12 | 확정 | 편성 단일 실행(명세 F004 동시성): 기준 구현은 단일 인스턴스 전제, 애플리케이션 잠금(synchronized) + 트랜잭션 내 재확인. 분산 환경은 범위 밖(명세 8.3.3) |
| 4-13 | 확정 | 스케줄 주기(명세 9.1 "구현 확정"): 60초 고정 지연(fixedDelay) 폴링 |
| 4-14 | 확정 | 기준 측정 행의 linked_date = 등록 시점 날짜(명세 7.1.4 그대로), 챌린지 시작 전 제출 거부(E-SUB-BEFORE-START) |
| 4-15 | 확정 | 멀티파트 업로드 한도 10MB(명세 12.3 미확정 → Wherehouse와 동일 값으로 확정) |
| 4-16 | 확정 | 30일 집계의 "그 종목" 판정: 제출이 속한 챌린지의 exercise가 대상 챌린지의 exercise와 같은 확인 완료 제출(기준 측정 행 포함, 명세 7.1.4) |
| 4-17 | 확정 | "목표 빈도는 하루 상한 안"(F002) 판정: 두 주기 방식 모두 하루 1회 이하를 전제하므로 '주 며칠' 간격이 7 초과인 경우만 E-APP-FREQ-OVER-CAP으로 거부 |

## 5. 미해결·추후 항목

- 인증샷 파일을 내려주는 GET 엔드포인트(팀원 확인 화면용)는 단계 5 UI 설계 때 추가 예정.
- 순위 산출은 기준 구현에서 확인 완료 제출을 DB 집계로 계산(명세 8.3.3·10.3 미확정 유지). 계측 후 자료구조·캐시 재결정.
- 팀 편성 r값·30일 윈도우 길이는 잠정값(명세 12.3). 계측 후 재확정.
