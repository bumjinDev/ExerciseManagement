# 권한 체계 설계안 (승인 요청)

작성일: 2026-07-07
근거: 클로드코드 작업지침서 1장("권한 체계는 이 프로젝트 도메인에 맞춰 새로 설계하고, 정하기 전에 보고"),
설계 명세서 3.1(인증 방식)·6장(API별 접근 조건)·8.2.2(확인 권한 검사)

## 1. 설계 원칙: 2계층 분리

Wherehouse의 권한 구성은 차용하지 않는다. 이 프로젝트의 접근 제어는 성격이 다른 두 판정으로 나뉜다.

| 계층 | 판정 내용 | 판정 근거 | 구현 위치 |
|------|-----------|-----------|-----------|
| 1계층: 인증 게이트 | 로그인한 사용자인가 | JWT 쿠키 (정적) | Spring Security 필터체인 |
| 2계층: 도메인 인가 | 그 챌린지의 참가자인가, 제출자와 같은 팀인가, 본인 제출이 아닌가 | PARTICIPATION·TEAM 등 DB 상태 (동적) | 각 도메인 Service |

**분리 이유**: "참가자·팀원"은 챌린지마다, 편성 시점마다 달라지는 동적 관계라 정적 역할(ROLE)로 표현할 수 없다.
Spring Security에는 "로그인 여부"만 맡기고, 관계 기반 판정은 설계 명세서가 이미 도메인 규칙으로 정의한 곳(8.2.2 권한 검사,
6.6/6.7 참가자 검증)에서 명세서의 에러 코드로 처리한다.

## 2. 역할(Role) 구성

- **ROLE_USER 단일 역할** 유지 (가입 시 자동 부여, 단계 2 구현 그대로).
- 요구사항·설계 명세서에 관리자 전용 기능이 없으므로 ROLE_ADMIN은 지금 만들지 않는다.
  운영 기능(예: 기본 상금풀 관리)이 생기면 그때 역할을 추가한다 — USERENTITY_ROLES 구조가 다중 역할을 이미 지원하므로 스키마 변경 없이 가능.

## 3. 경로별 접근 매트릭스

### 3.1 회원·인증 (단계 2 구현 유지)

| 경로 | 메서드 | 1계층 |
|------|--------|-------|
| /login, /logout | POST | 공개 |
| /members/join | POST | 공개 |
| /members/loginSuccess, /members/edit | GET/POST | 인증 필요 |

### 3.2 챌린지 도메인 (단계 3 신규, 전부 `/api/**` 필터체인 하나로 구성)

| 경로 | 메서드 | 1계층 | 2계층 도메인 인가 (명세서 에러 코드) |
|------|--------|-------|--------------------------------------|
| /api/challenges | GET | **공개** (6.5: 요청자 식별 불필요) | — |
| /api/challenges/{id} | GET | **공개** | 404 E-CHL-NOT-FOUND |
| /api/challenges | POST | 인증 필요 | 성립 제약 검사 (E-REG-...) |
| /api/challenges/{id}/participations | POST | 인증 필요 | 모집 기간(E-APP-NOT-RECRUITING), 중복 신청 차단(7.1.3 유니크), 잔액(E-APP-INSUFFICIENT-BALANCE) |
| /api/challenges/{id}/participations/baseline-measurement | POST | 인증 필요 | 기준 측정 요청 상태 확인(E-APP-NO-PENDING-MEASUREMENT) |
| /api/challenges/{id}/submissions | POST | 인증 필요 | 그 챌린지의 진행 중 참가자인지, 기계 검증 5종(8.2.1) |
| /api/submissions/{id}/confirmations | POST | 인증 필요 | 같은 팀(E-CFM-NOT-TEAMMATE), 본인 금지(E-CFM-SELF-CONFIRM), 종착 상태(E-CFM-ALREADY-TERMINAL) — 8.2.2 순서 그대로 |
| /api/challenges/{id}/rankings | GET | 인증 필요 | 참가자 검증(E-RNK-NOT-PARTICIPANT), 시작·종료 상태(E-RNK-NOT-STARTED) |
| /api/challenges/{id}/submissions/me | GET | 인증 필요 | 참가자 검증(E-SUB-NOT-PARTICIPANT) |
| /api/deposits (제안, 4장 참조) | POST/GET | 인증 필요 | 본인 잔액만 조작·조회 |

### 3.3 구현 방식

- SecurityConfig에 `/api/**` 필터체인 1개 추가: `GET /api/challenges`, `GET /api/challenges/*`만 permitAll, 나머지 authenticated.
  기존 `JwtAuthProcessorFilter` 재사용, 401은 `ApiAuthenticationEntryPoint`, 403은 `JwtAccessDeniedHandler`(모두 단계 2 산출물).
- 요청자 식별: JWT 클레임의 userId(= MEMBERTBL.id)를 SecurityContext에서 꺼내 참가자·팀원 판정에 사용.
- 2계층 검증 실패는 명세서 6.8 에러 포맷 `{"errorCode": "...", "message": "..."}`으로 응답 (챌린지 도메인 전용 @RestControllerAdvice 신설).

## 4. 함께 확인받을 항목 (명세서에 계약이 없는 부분)

1. **예치 충전 API 계약(F010)**: 명세서 6장에 엔드포인트가 없다. 다음을 제안한다.
   - `POST /api/deposits` — 바디 `{"amount": 50000}` (0보다 큰 정수), 응답 200 `{"balance": <갱신 잔액>}`
   - `GET /api/deposits/me` — 응답 200 `{"balance": ..., "challengeDeposits": [{"challengeId": ..., "amount": ..., "state": "진행 중|환급|차감|무산 반환"}]}`
2. **기본 상금풀(base_prize_pool) 출처**: 등록 요청 DTO에 없는 값(운영이 넣는 값)이므로,
   챌린지 등록 시 시스템 설정 프로퍼티(`app.base-prize-pool`, 잠정 500,000원)를 저장하는 방식을 제안한다.
3. **팀 편성 합 상한 r**: 미확정(잠정 설정값). 프로퍼티(`app.formation-sum-cap-percent`, 잠정 10%)로 두고 계측 후 재확정 TODO를 남긴다.

승인되면 이 문서를 기준으로 단계 3 SecurityConfig와 도메인 검증을 구현한다.
