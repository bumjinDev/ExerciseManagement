# 프로젝트 목적과 방향성

> 2026-06-09 · 이전 메모와 논의 반영본

---

## 1. 목적과 위치

- 새 프로젝트를 기획한다. 범위는 "작업 가능한 최대한의 범주"가 아니라, 다섯 계층이 맞물려 동작하는 통합된 단일 시스템으로 잡는다. 넓이는 학습 트랙을 늘려서가 아니라 하나의 시스템을 통합·유지하는 데서 나온다.
- 포트폴리오 목적은 유지하되 비중을 줄이고, 하고자 하는 시스템·기술 학습의 비중과 맥락을 키운다.
- 1차 목적은 장기 취업 중심이다. 단기 취업에 필요한 요소도 포함한다.
- 이 프로젝트는 향후 1~3개월 구직의 직접 레버가 아니다. 그 기간 면접 자산은 기존 Wherehouse·ChatService다. 이 프로젝트는 장기 학습·유지 수단이다.
- 기간(데드라인)은 선정 기준에서 제외한다. 기간보다 주제·도메인이 원하는 방향인지가 우선이다.

## 2. 기존 프로젝트에서 느낀 문제와 해소 방향

- 미취업 상태라 도메인 지식과 실제 비즈니스 맥락이 없었고, 그 결과 단순 기능 추가와 제한된 기술만 담겼다. 단조로운 비즈니스 맥락 중심 개발은 하지 않는다.

## 3. 주제 선정 — 도메인 후보와 설계 기준

프로젝트 형태는 웹으로 한정한다

후보 선정 기준 (도메인이 다음을 모두 만족해야 함):
- 4번 다섯 계층을 하나의 통합 시스템 안에서 자연스럽게 요구할 것.
- 기존 강점(동시성=ChatService, 정합성·보상=Wherehouse, TLS, Redis)을 연장할 것.
- 알고리즘·자료구조 의사결정 밀도 — 도메인 코어의 접근 패턴이 자료구조 선택을 강제할 것(예: 정렬 유지+최솟값/범위 접근→TreeMap, 동시 접근+정렬→ConcurrentSkipListMap, 유계 큐→백프레셔와 거부 정책). 부가 기능에 자료구조를 끼워 넣는 방식은 인정하지 않는다.
- CPU 바운드 작업의 자연 발생 — 대량 집계, 파생값 대량 재계산, 매칭·탐색 계산, 대량 필터링 중 하나 이상이 비즈니스 요구로 존재할 것. 도구(ForkJoinPool 등)를 목표로 두고 도메인을 고르지 않는다.
- 의사결정 분기 다양성 — 다섯 계층 각각에서 "왜 X가 아니라 Y" 분기가 코어 경로에 자연 발생할 것. 분기를 스택 추가로 만들지 않는다.

설계 기준 (박람회 피드백을 요구사항으로 둠):
- 아키텍처·의사결정이 먼저 보일 것.
- 요구사항 수집·분석·기획 과정이 보일 것.
- 기술 선택 근거를 논리적으로 설명할 것(왜 X가 아니라 Y).
- 테스트 수행 흔적을 남길 것.
- AI 에이전트가 대체하기 어려운 시스템 내부 설계 판단을 담을 것.

## 4. 포함할 학습·기술 범주 — 다섯 계층, 통합

**공통 원칙** — 각 계층을 인프라 제품 내부 수준으로 밑바닥부터 구현하는 것이 목표가 아니다. 시스템을 구현해 운영하면서 비즈니스 맥락에서 실제로 발생하거나 필요해진 것을 디버깅하고 의사결정으로 잇는다. 깊이는 주요 피드백 2개(박람회)가 인정하는 선 — 논리적 설명과 의사결정 과정 — 까지이며 취업 범위를 넘지 않는다.

**동시성**
- 기존에는 격리된 단일 레지스트리 수준이었다. 이번엔 전체 시스템에 통합한다.
- CPU 바운드 구간이 확인되면 다음 의사결정 체인을 따른다.
  1. 바운드 종류 계측 — 부하 중 스레드 덤프의 RUNNABLE 비중, CPU 사용률 대비 처리량으로 I/O 바운드와 구분한다.
  2. 병렬화 모델 선택 — 분할 정복형이고 하위 작업 크기가 불균등하며 논블로킹이면 ForkJoinPool, 독립적이고 균등한 작업이면 ThreadPoolExecutor.
  3. 분할 임계값 결정 — 태스크 생성 오버헤드와 계산량의 교차점이므로 측정으로 정한다.
  4. 운영 격리 — parallelStream의 전역 commonPool 경합을 확인하고 전용 풀 분리와 코어 배분을 결정한다.
- ForkJoinPool은 이 체인의 결론으로만 등장하며 목표로 두지 않는다.

**네트워크·프로토콜**
- 기존에는 분석에 그쳤다. 운영 중 발생하는 네트워크·프로토콜 문제를 다룬다.

**저장소·파일시스템**
- 데이터 영속·파일 처리에서 발생하는 부분.

**리눅스**
- 프로젝트가 동작하는 환경으로서의 운영 레벨. 빌드·배포·systemd 서비스 등록·프로세스 기동/상태/재시작·네트워크 설정·로그·자원 한계(ulimit/FD)·장애 대응.
- 기존 Ubuntu·systemd·certbot 운영의 연장.
- 커널·시스콜 관측(strace·perf·eBPF)은 부차적이며 특정 문제가 실제로 요구할 때만 본다.

**알고리즘·자료구조**
- 수준: 코딩 테스트가 요구하는 수준(예: 레드-블랙 트리, 큐잉 구조 등).
- 기본형은 선택·운용·계측 — JDK 구현체를 비교해 선택하고, 내부 구조를 선택 근거로 설명하며, 선택의 효과를 계측한다.
  - 선택 예: TreeMap vs HashMap vs ConcurrentSkipListMap, 블로킹/논블로킹/우선순위 큐와 거부 정책.
  - 근거 예: 동시성 정렬 맵이 레드-블랙 트리가 아니라 스킵 리스트인 이유 — 트리 리밸런싱은 락 분할이 어렵고 스킵 리스트는 노드 단위 CAS가 가능하다.
  - 무구현 관측 지점 예: HashMap 버킷의 레드-블랙 트리 전환.
- 예외는 제한적 직접 구현 — 핵심 자료구조 1~2개에 한해 허용하되, 산출물은 구현 자체가 아니라 표준 구현과의 벤치마크 비교다(기존 동시성 4기법 비교 테스트와 동일한 방법론).
- 경계: 자료구조는 도메인 코어 경로에 강제로 존재해야 하며 부가 기능에 끼워 넣지 않는다.
- 효과: 코딩 테스트로 학습하는 자료구조에 실제 적용처를 만들어 반복 망각도 함께 막는다.

## 5. 언어·스택

- Java 기반 Spring Boot 웹 프로젝트로 한다(취업 방향). 프로젝트 형태는 웹이 우선이나 앱한정한다.
- C 등 시스템 언어는 처음부터 트랙으로 세우지 않는다. 도메인·개발 중 실제로 필요해지면 그때 자연히 포함한다(필요 주도).
- "언어는 도구일 뿐"의 정리: 목적이 시스템을 이해하는 것이라면 아래 계층은 C를 작성해서가 아니라 관찰·운영해서 이해한다. 따라서 이 명제는 여기서 C를 빼는 근거다. 도구(Java)는 그대로 두고 관찰·운영을 시스템 쪽으로 내린다.

## 6. 학습 방식 — 반복 망각의 해소

- 책·강의를 읽기만 하고 실제 프로젝트에 반영하지 않아 반복적으로 잊었다.
- 해소: 격리된 학습이 원인이므로, 다섯 계층이 맞물리는 하나의 시스템에 녹여 넣고 계속 관리·유지한다. 좁히기가 아니라 통합과 유지가 해법이다.

## 7. 학습 수준 기준 / 미결 / 다음 단계

- 학습 수준 기준: "실제로 만들어 동작·운영시키는 수준"을 기준으로 잡는다. 리눅스는 취업 시장 운영 레벨, 시스템은 직접 구현 후 계측·운영하는 수준이다.
- 미결(현재 선결 결정): 도메인 확정. 도출된 후보 5개(3번)를 추가된 선정 기준(자료구조 의사결정 밀도·CPU 바운드 자연 발생·분기 다양성)으로 재평가한 뒤 그중에서 확정한다.
- 다음 단계: 도메인 확정 후 그 도메인의 요구사항·기획부터 시작한다(대상 사용 맥락·기능 범위 등).
- 포트폴리오 구성: 새 인프라성 웹 프로젝트가 ChatService의 동시성 코어와 겹친다. (1) 3번째로 추가, 또는 (2) ChatService를 대체. 도메인 확정 후 결정한다.

## 8. 개인적·심리적 동기와 경계

- 정량적 학벌·나이·대외활동 부족에서 오는 자괴감·나태함·무력함·불안이 있다. 완전히 떨칠 수는 없어도 가능한 선까지 보완하고 납득하는 것을 목표로 한다. 이 제약은 실재한다.
- 경계: 다만 프로젝트가 이 문제를 직접 해결하지는 않는다. 서류 필터는 프로젝트를 보기 전 정량 지표로 작동한다(6번·박람회 동일 지적). 프로젝트는 면접 단계에서 역량 차이를 만들지, 서류 단계를 통과시키지는 않는다.
- 분리: "역량·유지를 위한 프로젝트"와 "서류 통과를 위한 행동"을 분리한다. 서류 통과 레버는 채널·추천·회사 선택 쪽이다(점핏·원티드·직접 지원, 자소설닷컴 회피).
