## 1. 공통 원칙

- 스택은 설계명세서 11.1을 따른다: Java 17, Spring Boot 3.3.5, Spring Security, Oracle Database.
- DB는 이 프로젝트 전용 Oracle 계정과 스키마만 사용한다. Wherehouse의 계정·스키마·테이블에 접근하지 않는다.
- DB 반영 방식: `ddl-auto: none`. 애플리케이션이 스키마를 만들지 않는다. 계정 생성, 테이블 생성, 시드 데이터 등 모든 SQL은 프로젝트 내 `sql/` 디렉토리에 실행 순서가 드러나는 파일명(`01_create_user.sql`, `02_member_tables.sql`, ...)으로 작성해서 전달한다. 실행은 내가 로컬 Oracle에 직접 한다. 단계가 진행되며 테이블이 늘어나면 새 번호의 파일로 누적한다.
- Redis는 기준 구현에 넣지 않는다. 순위 누적 저장 전략과 캐시는 계측 후 결정하는 미확정 항목이다(설계명세서 8.3.3, 10.3). 회원 로직 이식 시 Wherehouse의 `RedisHandler` 의존은 제거한다.
- 권한 체계는 Wherehouse 것을 차용하지 않는다. 이 프로젝트 도메인에 맞춰 새로 설계하고, 정하기 전에 내게 안을 보고한다.
- 문서에서 확인되지 않는 값은 추정으로 채우지 말고 질문한다.
- 설계명세서가 "구현 단계에서 확정" 또는 "계측 후 결정"으로 남긴 항목을 임의로 확정하지 않는다.

## 2. 단계별 작업 (순서대로 진행)

### 단계 1: DB 준비

회원 관리 개발 전에 먼저 수행한다.

1. 전용 Oracle 계정 생성 SQL 작성: 계정 생성, 필요한 최소 권한 부여(CREATE SESSION, CREATE TABLE, CREATE SEQUENCE 등), 테이블스페이스 지정. 계정명은 시작 전에 내게 확인받는다.
2. 회원 관리 테이블 DDL 작성: 구현환경_참고.md 3.2의 구조(회원 테이블, 인증 테이블, 역할 테이블)를 기준으로 이 프로젝트에 맞게 작성한다.
3. 두 SQL을 `sql/`에 저장해 전달하고, 내가 로컬 반영을 확인해 줄 때까지 다음 단계로 넘어가지 않는다.

### 단계 2: 회원 관리 기능 (로그인·회원 가입·정보 수정)

Wherehouse의 `com.wherehouse.JWT`, `com.wherehouse.members` 패키지를 확인하고 동일한 기능으로 구현한다. 클래스 목록은 구현환경_참고.md 3.1에 있다.

이식 시 반드시 조정할 것 (구현환경_참고.md 3.3):

- `MemberService`, `LoginFilter`의 `RedisHandler` 의존 제거.
- JWT 유효 기간은 테스트를 위해 **2주(14일)** 로 설정한다. 하드코딩하지 말고 설정 프로퍼티로 분리하고, 운영 전 재확정 대상임을 TODO 주석으로 남긴다.
- 로그인 성공 리다이렉트 경로(`/wherehouse/members/loginSuccess`)를 이 프로젝트 경로로 변경.
- 쿠키 `setSecure(false)`는 로컬 개발 기준으로 두되 프로파일로 전환 가능하게 한다.
- JWT 서명 키는 환경 변수 `JWT_SECRET_KEY`로 주입받는다. 키 생성 방법을 내게 안내한다.

완료 기준: 회원 가입, 로그인(JWT 쿠키 발급), 로그아웃, 정보 수정이 로컬에서 동작하고 확인 방법을 보고한다.

### 단계 3: 백엔드 본 기능 개발

요구사항 명세서와 설계명세서 v1.4를 따라 서버 백엔드를 개발한다.

- 구현 범위와 계약은 설계명세서 6장(API), 7장(데이터 모델), 8장(비즈니스 로직), 9장(스케줄), 10장(실시간)을 따른다.
- 신규 테이블 DDL은 단계 1과 같은 방식으로 `sql/`에 파일로 전달하고 내 반영 확인 후 진행한다.
- 요청자 식별은 단계 2의 인증 체계를 사용한다(설계명세서 3.1).

### 단계 4: API 테스트 명세서 (Postman)

기본 기능에 대한 API 테스트를 Postman으로 진행할 수 있게 테스트 명세서를 작성해 전달한다.

- 포함 내용: 엔드포인트별 메서드·URL·요청 헤더와 바디 예시, 기대 응답(정상·에러 코드), 실행 전제(로그인해서 JWT 쿠키를 얻는 절차, 테스트 데이터 준비 SQL).
- Postman에 바로 임포트할 수 있는 컬렉션 JSON 파일도 함께 제공한다.
- 명세서의 에러 코드는 설계명세서의 에러 코드 체계와 일치해야 한다.

### 단계 5: UI 개발

- 시작 전에 화면 목록과 UI 기술(서버 렌더링 JSP 방식 또는 정적 페이지 + API 호출 방식)을 내게 확인받고 진행한다.
- 순위·현황 화면은 설계명세서 6.6의 폴링 방식(화면의 JS가 API를 주기 호출)을 따른다.

## 3. 진행 규칙

- 각 단계 완료 시 보고: 만든 것, 확인 방법, 다음 단계에서 필요한 결정 사항.
- 단계 1과 단계 3의 SQL은 내 로컬 반영 확인이 게이트다. 확인 전에 해당 테이블을 전제로 한 코드를 완성품으로 보고하지 않는다.
- 내게 확인받을 값: Oracle 계정명·비밀번호, 서버 포트·컨텍스트 경로, UI 기술 선택.
